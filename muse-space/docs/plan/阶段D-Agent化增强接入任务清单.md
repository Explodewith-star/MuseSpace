## 3. 可执行任务清单

### D1-1 固化 Agent 运行时契约

目标：统一所有 Agent 的执行方式，避免后续每个 Agent 都单独写一套流程。

任务：

- 统一 `IAgentRunner`、`AgentDefinition`、`IAgentTool`
- 统一 Agent 输入输出结构
- 统一工具调用协议与 JSON Schema 表达
- 统一失败、取消、无工具模式的执行分支

实现思路：

- 这一层只负责运行时，不绑定具体业务 Agent
- 单轮任务仍然保留在 Skill 体系，避免 Agent 泛化过度

验收：

- 可运行无工具 Agent 和带工具 Agent

状态建议：已完成。

### D1-2 可观测与 Feature Flag

目标：让 Agent 上线后可追踪、可灰度、可关闭。

任务：

- 统一 `AgentRun` 持久化
- 记录 token、耗时、状态、错误信息
- 引入总开关和单 Agent 开关
- 将日志与 AgentRunId 关联

实现思路：

- 没有可观测就不要继续扩 Agent 数量
- 先保证“看得见”，再做“跑得快”

验收：

- 每次 Agent 执行都有落库记录
- 可以按配置关闭某个 Agent 而不影响主流程

状态建议：已完成主干。

### D1-3 角色提取 Agent 归一化

目标：把现有角色提取能力变成标准样板，供后续 Agent 复用。

任务：

- 确认 `CharactersController` 已通过 `IAgentRunner` 调用角色提取 Agent
- 固化输入结构、输出结构、失败处理方式
- 提炼可复用的 Prompt 与 schema 校验模式

实现思路：

- 这一项不是业务终点，而是后续一致性 Agent 和大纲 Agent 的模板

验收：

- 角色提取链路稳定且可观察

状态建议：已完成。

### D1-4 统一任务中心与建议审核层

目标：让后续所有菜单 Agent 都跑在统一的状态流和审核流上，而不是页面各自实现一套流程。

任务：

- 定义统一任务模型，覆盖提取、规划、校验、改写等异步 Agent 任务。
- 定义统一建议模型，例如 `AgentSuggestion`，承载候选角色、候选规则、候选大纲、候选章节等结果。
- 明确建议状态流转：建议态、待确认态、已应用态、已忽略态。
- 为前端提供统一任务列表、结果查看、批量接受 / 忽略的接口约定。

实现思路：

- 菜单 Agent 化的关键不是“多一个按钮”，而是统一任务中心和统一建议审核机制。
- 没有这一层，后续大纲、角色、世界观、文风的 Agent 结果都会变得碎片化。

验收：

- 至少两类不同 Agent 结果可以通过同一套建议状态流转进行查看和确认。

状态建议：应作为 D2 / D3 之前的基础补齐项。

### D2-1 世界观一致性 Agent

目标：让已存在的 `WorldRule` 真正参与生成后的审查。

任务：

- 设计 `GetWorldRulesTool`
- 设计输出结构，例如冲突片段、严重等级、修正建议
- 异步挂到草稿生成后处理流程
- 设计前端一致性面板展示冲突结果

实现思路：

- 这是“生成后守护”，不是“生成前阻断”
- 失败必须可降级为“不显示结果”，而不是影响草稿生成成功与否

验收：

- 带明显世界观冲突的样例可识别出大部分问题

状态：✅ 已完成。`ConsistencyCheckJob` 在草稿 / 大纲 完成后自动触发，结果进 `WorldRuleConsistency` / `OutlineConsistency` 类目；前端世界观页顶部 `PendingSuggestionPanel` 直接展示待办。

### D2-2 角色一致性 Agent

目标：检测角色行为、身份、关系和状态是否与已有角色卡冲突。

任务：

- 设计 `GetCharacterCardTool`
- 如有需要，增加章节搜索工具辅助定位上下文
- 统一与世界观一致性 Agent 的 annotation 结构
- 前端复用一致性面板

实现思路：

- 角色一致性和世界观一致性应共用一套展示模型
- 不要分别做两套前端和两套存储结构

验收：

- 人工构造的角色设定冲突可被检测并给出建议

状态：✅ 已完成。`CharacterConsistencyCheckJob` 在 `ChapterDraftJob` 完成后链式入队，结果进 `CharacterConsistency` 类目，角色页顶部扫描入口与待办面板均已接入。

### D3-1 大纲规划 Agent

目标：从“帮用户写一段”扩展到“帮用户搭建长篇结构”。

任务：

- 定义输入表单：题材、主角、冲突、字数、参考风格
- 支持新作规划、续写规划、番外规划、局部补全四类输入模式
- 设计多步输出：三幕、卷级结构、章纲、约束与线索草案
- 提供一键导入章节树能力
- 支持局部重生成功能，例如重做某一卷而不影响全局

补充任务：

- 定义 `OutlinePlan`、`OutlineVolume`、`OutlineChapter` 等数据结构或等价模型。
- 让大纲结果优先以结构化 JSON / DTO 返回，再进入建议审核层，而不是直接写入正式章节表。
- 增加大纲一致性校验，检查与现有角色、世界观、章节状态是否冲突。

实现思路：

- 大纲规划是第一个真正适合多步 Agent 的业务场景
- 先输出结构化 JSON，再入库，不要让 Agent 直接写自由文本进章节表
- 大纲不是单纯的文本计划，而是后续章节树、伏笔、草稿生成的骨架

验收：

- 能生成结构合理的长篇章节树，并允许局部改写
- 能基于已导入原著生成续写或番外大纲，而不是只支持从零规划

状态建议：未开始。

### D3-1.1 大纲功能最小产品边界

目标：在大纲功能第一版就明确边界，避免一开始做成“会说很多但无法落库”的文本生成器。

任务：

- 前端提供三类模式：新作规划、续写规划、番外规划；局部补全可作为三类模式下的局部操作。
- 输出至少稳定覆盖四层：大纲摘要、卷级结构、章级结构、线索与约束草案。
- 所有结果先进入建议审核层，再决定是否导入正式章节树。
- 支持“整份接受”“单卷重做”“单章编辑后导入”三类基本交互。

实现思路：

- 大纲功能第一版先解决“结构化、可审阅、可导入”，不追求一次覆盖所有长篇规划高级能力。
- PlotThread、长期版本管理和复杂关系推理留给阶段E逐步增强。

验收：

- 用户能基于一句话目标得到一份可编辑、可局部重做、可导入章节树的大纲候选结果。

#### D3-1 遗留 UI 优化项（随 D3 推进时顺带处理）

- **章节编号空洞**：删除某章后编号出现空洞（如 1,2,3,5,6）。建议在大纲页或章节管理页提供"重排编号"操作，将所有章节 Number 按顺序重新赋值。`OutlineDetail.vue` 已有 UI 占位，后端接口待补。

### D3-2 菜单 Agent 化与统一入口

目标：让概览、章节、角色、世界观、文风、草稿生成、原著导入逐步拥有统一风格的 Agent 能力入口。

任务：

- 定义统一交互原则：一句话目标 + 补充约束 + 结果审核。
- 明确每个菜单当前最适合的 Agent 类型：提取型、规划型、守护型、执行型。
- 原著导入页提供“导入后自动提取候选资产”的任务入口。
- 概览页提供项目摘要与下一步建议入口，章节页提供大纲规划与局部补全入口。

实现思路：

- 不把每个页面做成单独聊天机器人，而是让每个菜单成为项目资产工作台。
- 所有菜单共享 Agent 运行时、任务中心、建议审核层和工具系统。

验收：

- 至少三个菜单页能通过统一交互方式触发不同类型的 Agent 任务。

状态建议：未开始，可与 D3-1 并行推进。

### D3-3 原著导入后的自动提取流水线

目标：让“原著导入完成”不再只是索引完成，而是正式成为后续 Agent 能力的起点。

任务：

- 导入完成后按任务链依次触发候选角色提取、候选世界观提取、文风画像、章节摘要整理。
- 所有提取结果默认进入统一建议审核层，而不是直接写入正式资产表。
- 在原著导入页或统一任务中心展示这些后处理任务的状态、结果和确认入口。

实现思路：

- 这条流水线是“菜单 Agent 化”的真正起点，能把原著从被动存储转成可持续消费的项目资产。
- 第一版不追求提取全量知识，只优先提取最直接服务创作的角色、设定、文风和章节摘要。

验收：

- 用户导入一本原著后，能够在统一任务中心看到至少三类候选资产被自动提取并待确认。

### D4-1 伏笔追踪与风格 Reviewer

目标：让 Agent 从“生成辅助”进入“长期创作协作”。

任务：

- 定义 `PlotThread` 数据模型

实现思路：

验收：

- 伏笔能被记录、提示、回收
- Reviewer 可稳定产出可操作建议

1. 先完成并稳定 D1
2. 优先补统一任务中心与建议审核层，避免后续菜单 Agent 化碎片化

## 5. 高级能力引入时机

### Hybrid 检索作为 Agent 工具

引入时机：D2 后半段开始。

具体步骤：

1. 一致性 Agent 先用现有结构化工具和基础检索工具
2. 当角色、世界观或章节引用需要更精准召回时
3. 再把 Hybrid 检索包装为 Agent 工具

原因：Agent 的问题首先是“流程可控”，不是“检索多复杂”。

### GraphRAG 作为 Agent 背后能力

引入时机：D4 后期或阶段E。

适用场景：

- 伏笔追踪跨很多卷
- 角色关系和组织关系图变得复杂
- 需要跨实体、跨时间线查询时

在此之前，优先使用 PostgreSQL 的关系表和普通检索工具即可。

### Redis / 对象存储 / Worker

引入时机：不作为阶段D默认前置条件。

原因：阶段D的核心目标是把菜单 Agent 化和规划层跑通，而不是先进行工业化拆分。

### Multi-Agent

引入时机：阶段E后段。

原因：当前绝大多数场景单 Agent 加工具已经够用，过早拆多 Agent 会显著增加调试和成本复杂度。
