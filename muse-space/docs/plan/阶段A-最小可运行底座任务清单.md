# 阶段A：最小可运行底座任务清单

> 阶段目标：为记忆层和后续 Agent 能力补齐最小可运行底座，但不引入会显著提高复杂度的工业化组件。
>
> 当前状态：基本完成，可视为已落地底座阶段，后续仅做补缺和规范化收尾。

---

## 1. 阶段范围

- PostgreSQL 16 与 pgvector 可用
- 记忆相关基础实体与迁移可用
- 本地文件存储目录和接口可用
- 现有生成链路可在该底座上正常运行

---

## 2. 可执行任务清单

### A-1 开发环境标准化

目标：统一本地开发和换机时的数据库准备方式。

任务：

- 明确当前默认开发数据库方案是本地 PostgreSQL 还是远程托管库
- 明确数据库版本、扩展要求和初始库名约定
- 将数据库连接配置方式统一为 `appsettings.Development.json` 或 user-secrets
- 补一份最小启动说明，至少覆盖建库、启扩展、跑迁移、启动 API

实现思路：

- 这一步只解决“任何开发机都能跑起来”，不做多环境发布编排
- 若未来引入 Docker Compose，也只作为开发便利工具，不改变阶段A的架构判断

验收：

- 新开发机可在半小时内完成数据库准备并跑通项目

状态建议：已部分完成，建议补文档和初始化脚本。

### A-2 PostgreSQL 扩展与 Schema 基线

目标：确保记忆检索和后续审计扩展有明确数据边界。

任务：

- 确认 `vector` 扩展已作为数据库前置要求
- 确认 `memory` schema 的使用范围
- 如后续需要，补 `audit` schema 占位而不急于落表
- 检查迁移是否对 schema 依赖有明确表达

实现思路：

- 当前阶段只需要保证 pgvector 和记忆相关表可正常工作
- 不要在阶段A引入复杂多 schema 权限体系，避免维护负担提前出现

验收：

- 数据库初始化后可直接应用迁移，不需要手工补表

状态建议：已基本完成，建议做一次迁移链路回归验证。

### A-3 本地文件存储抽象

目标：给原著导入提供最小可替换存储层。

任务：

- 明确原始文件和解析产物目录结构
- 确认数据库只存逻辑文件 Key，不存物理绝对路径
- 保持 `IStorageService` 或等效抽象在基础设施层统一管理
- 补充失败时的错误处理约定，例如文件已写入但数据库未提交时如何清理

实现思路：

- 阶段A只做“可替换”，不做“云原生”
- 只有在单机文件存储成为协作瓶颈时，才进入阶段E替换为对象存储

验收：

- 上传后的文件可读回
- 删除、重传、失败清理流程行为清晰

状态建议：已基本完成，建议补异常回滚策略说明。

### A-4 记忆实体与迁移规范

目标：让后续阶段的新增表和数据关系都通过可追踪迁移进入项目。

任务：

- 确认 `Novel`、`NovelChunk`、`NovelChunkEmbedding` 等基础实体已纳入 DbContext
- 确认迁移命名规范、创建方式、应用方式
- 明确禁止手改数据库结构
- 明确换机或 CI 初始化时的迁移执行顺序

实现思路：

- 阶段A的核心不是“表多”，而是“今后所有表都能按统一规范增长”
- 为阶段E预留迁移扩展空间，但不提前设计复杂版本治理

验收：

- 全新数据库执行迁移后可直接启动系统

状态建议：已完成主干，建议补 README 或 docs 中的操作约定。

## 3. 实现顺序

1. 固化数据库准备流程
2. 校验扩展和 schema
3. 校验本地文件存储抽象
4. 统一迁移规范和初始化流程

---

## 4. 本阶段不该引入的高级能力

以下能力在阶段A不应引入：

- Hybrid RAG
- GraphRAG
- Neo4j
- Redis
- 独立 Worker
- Multi-Agent 编排

原因：阶段A只解决“能跑”和“可持续演进”的底座问题，引入这些能力只会增加调试面。

---

## 5. 高级能力的真正引入时机

### Hybrid RAG

引入时机：阶段C 后半段。

判断条件：纯向量检索已经稳定接入生成链路，但命名词、专有名词、设定条目召回不稳定。

### GraphRAG / Neo4j

引入时机：阶段E 中后段。

判断条件：实体关系和跨章节线索查询已明显超出 PostgreSQL 关系表 + 向量检索的可维护范围。

---

## 6. 阶段完成定义

- 新开发机可快速完成初始化
- 记忆相关基础表和迁移链路稳定
- 文件存储抽象稳定可用
- 现有 Skill 主链路不受影响
