# C-5 大纲规划输入基线

> 阶段C任务清单 C-5 的产出。明确大纲规划 Agent（`OutlinePlanAgentDefinition` / `OutlinePlanJob` / `RegenerateOutlineVolumeJob`）所消费的上下文边界，避免后续与草稿生成（`SceneDraftSkill` + `StoryContextBuilder`）的检索方式互相串扰。
>
> 关联代码：
> - `src/MuseSpace.Infrastructure/Jobs/OutlinePlanJob.cs`
> - `src/MuseSpace.Infrastructure/Jobs/RegenerateOutlineVolumeJob.cs`
> - `src/MuseSpace.Application/Services/Agents/OutlinePlanAgentDefinition.cs`
> - `src/MuseSpace.Infrastructure/Story/StoryContextBuilder.cs`（草稿生成上下文，仅作对比参考）

---

## 1. 设计原则

1. **规划查询 ≠ 生成查询**：大纲规划关心"全局结构与约束"，草稿生成关心"当前场景的文风与细节"。两者对上下文颗粒度的要求不同，因此**不复用 `StoryContextBuilder`**。
2. **结构化资产优先于原著片段**：大纲规划阶段优先使用结构化数据（角色卡、世界观规则、已有章节摘要），原著文本片段在大纲阶段仅作为"风格 / 题材参考"的可选输入，**不进入主输入**。
3. **进建议审核层，不直接落库**：所有大纲产出先以 `AgentSuggestion(category=Outline)` 形式进入审核层，经用户确认后才写入 `chapters` 表。

---

## 2. 当前已实现的输入构成

`OutlinePlanJob.ExecuteAsync` 当前对每条规划请求组装的上下文如下（按顺序）：

| 段落 | 来源 | 字段 | 是否必填 |
|---|---|---|---|
| 项目角色清单 | `ICharacterRepository.GetByProjectAsync` | `Name` / `Role` / `PersonalitySummary` | 有则写入 |
| 世界观规则清单 | `IWorldRuleRepository.GetByProjectAsync` | `Title` / `Description` / `IsHardConstraint` | 有则写入 |
| 已有章节清单 | `IChapterRepository.GetByProjectAsync` | `Number` / `Title` / `Summary or Goal` | 仅在 `continue` / `extra` 模式 |
| 用户目标 | 接口入参 | `goal`、`chapterCount`、`mode` | 必填 |

输出固定为 JSON：`{ volumes: [{ title, summary, chapters: [...] }] }`，再由 Job 解析后写入建议表。

---

## 3. 三类规划模式的输入差异（基线约定）

| 模式 | 主要输入 | 不应包含的输入 | 起始章节号 |
|---|---|---|---|
| `new`（全新规划） | 项目摘要 + 角色 + 世界观 + 用户目标 + 章节数 | 已有章节（应为空，否则视作 continue） | 1 |
| `continue`（续写扩展） | `new` 的全部 + **全部已有章节摘要**（按 `Number` ASC） | 与主线无关的番外卷 | `max(existing.Number) + 1` |
| `extra`（番外） | `new` 的全部 + 已有章节摘要（用作背景，不作主线延展） | 主线后续走向 | `max(existing.Number) + 1` |

> 当前实现已与该基线一致；`extra` 模式在 prompt 中显式告诉模型"主题应区别于主线"。

---

## 4. 单卷重做（`RegenerateOutlineVolumeJob`）的输入边界

- **应包含**：当前完整大纲建议（除目标卷外）、目标卷原内容、用户对该卷的修改意图、世界观与角色硬约束。
- **不应包含**：其他建议状态为 `Ignored` 的备选大纲；原著片段；风格画像。
- **目的**：让模型知道"这一卷在整本书的哪一段位置"，避免重做后的一卷与前后剧情失联。

---

## 5. 与草稿生成（`StoryContextBuilder`）的边界对比

| 维度 | 大纲规划（本文档） | 草稿生成（`StoryContextBuilder`） |
|---|---|---|
| 角色 | 全量角色清单（精简） | 涉及角色 4 张以内的完整卡 |
| 世界观 | 全量规则（含硬约束） | 优先 8 条（按 Priority） |
| 章节摘要 | 全部已有章节（仅 continue / extra） | 最近 3 章 |
| 风格画像 | **不输入** | `StyleRequirement` 全量 |
| 原著片段 | **不输入**（V1）；后续可作为题材参考 | top-5 召回 + 阈值 + 字符预算 |
| Scene 目标 | N/A | `SceneGoal` / `Conflict` / `EmotionCurve` |

---

## 6. 后续可演进项（不在本期范围）

1. **题材参考检索**：在 `new` 模式允许用户勾选"参考已导入原著的题材风格"，按 goal 检索 top-3 chunk 拼到 prompt 末尾，以"参考风格"段落形式注入；需要时再开。
2. **大纲一致性预检**：大纲生成后由 `ConsistencyCheckJob` 自动入队预检（已落地）。
3. **PlotThread 注入**：当 D4-1 伏笔追踪深化引入"未回收线索"提醒后，`continue` 模式可把高优先级未回收线索摘要追加到 prompt，让 Agent 主动回收。
4. **版本化**：阶段 E-3.5 涉及大纲版本与来源追踪，本文档不展开。

---

## 7. 验收标准（与 C-5 对应）

- 同一项目分别提交 `new` / `continue` / `extra` 三种模式时，得到的输入集合应符合本文档第 3 节的规约；可通过 `[OutlinePlan]` 日志或 `OutlinePlanJob` 调试断点确认。
- 单卷重做不会引入与目标无关的备选大纲。
- 大纲规划与草稿生成的检索 / 上下文构建路径互不污染。
