# 综合计划大纲

> 文档目的：整合《MuseSpace-记忆层实行计划精简plan》《初级记忆方案的实现》《业务过渡Agent实现计划》，形成一份面向 MuseSpace 当前阶段的综合实施思路文档。
>
> 适用范围：原著导入、基础记忆构建、续写与番外生成、Agent 化能力过渡、后续工业化扩展。
>
> 文档定位：作为后续分阶段开发、排期、验收和架构决策的统一纲要。各阶段的执行版任务清单已拆分到 `docs/plan/` 目录，本文件负责总览、顺序、状态和阶段边界。

## 1. 总体目标

MuseSpace 当前已经具备以下基础能力：

- `ISkill` + `SkillOrchestrator` 单步技能编排
- `SceneDraftSkill` 草稿生成链路
- `StoryContextBuilder` 上下文构建
- Prompt 模板体系
- Hangfire 后台任务
- SignalR 实时推送
- pgvector 向量存储底座
- `Character`、`WorldRule`、`StyleProfile`、`Chapter` 等核心业务数据
- 初步 Agent 运行时与 `AgentRun` 持久化能力

本综合计划的核心目标不是重新设计一套系统，而是在现有能力之上，按“先跑通主链路，再逐步增强自治能力，最后补工业化能力”的节奏推进，形成一条完整演进路径：

```text
原著导入
→ 文本切片
→ Embedding 与检索
→ 记忆注入续写 Prompt
→ Agent 介入一致性校验与规划辅助
→ 长期演进到更完整的记忆网络与创作协同体系
```

### 1.3 文档组织原则

为了避免计划文档持续膨胀、执行时仍需要重新理解，这套文档后续统一遵循“两层一附录视图”的组织方式：

- `综合计划大纲` 只负责全局路线、阶段边界、能力地图、技术引入条件、当前已完成能力回填。
- `阶段A` 到 `阶段E` 文档只负责当前阶段的执行范围、依赖关系、验收标准、技术判断和不该提前做的事。
- 菜单能力地图、阶段能力视图、技术引入判定表作为总纲中的固定视图维护，不在各阶段文档重复定义一套。

后续补文档时默认遵守以下边界：

- 总纲不展开到代码级步骤和实现细节。
- 阶段计划不重复书写全局背景和长期愿景。
- 技术引入条件优先在总纲定义一次，阶段计划只引用并细化本阶段判断。
- 当前已落地能力统一回填到总纲，阶段文档只维护本阶段完成度和缺口。

## 2. 核心原则

### 2.1 先跑通，再工业化

优先完成“导入原著 + 检索原著 + 辅助续写/番外”这一主场景，不在前期强制引入 Redis、Neo4j、对象存储、独立 Worker、Semantic Kernel、Multi-Agent 等额外复杂度。

### 2.2 复用现有体系，不重复造轮子

- 单轮生成继续走 `ISkill`
- 多步推理、工具调用、自治校验走 `IAgentRunner`
- LLM 能力统一复用现有 `ILlmClient` / `LlmProviderSelector`
- 长任务统一复用 Hangfire
- 实时状态统一复用 SignalR

### 2.3 业务价值优先于架构完整性

优先做能显著提升创作体验的能力：

- 原著检索辅助续写
- 世界观与角色一致性守护
- 大纲规划与长篇线索追踪

### 2.4 模块化演进

每个阶段都应满足：

- 可以独立开发
- 可以独立验收
- 失败可局部回退
- 不阻塞已有主流程

### 2.5 可观测与可降级

- 记忆链路要有导入状态、处理进度、失败原因
- Agent 链路要有运行记录、Token、成本、失败轨迹
- 增强能力失败时不应拖垮主生成流程

## 3. 统一架构思路

### 3.1 三层能力结构

综合三份方案后，MuseSpace 后续能力可以统一为三层：

#### 第一层：生成主链路层

负责直接面向用户完成续写、番外、草稿等核心产出。

- `SceneDraftSkill`
- `SkillOrchestrator`
- `StoryContextBuilder`
- Prompt 模板体系

这一层必须保持稳定，是所有增强能力的落点。

#### 第二层：记忆支撑层

负责提供可检索、可注入、可追踪的文本记忆和结构化上下文。

- 原著文件导入
- 文本解析与切片
- Chunk Embedding
- 向量检索与 Hybrid RAG
- 记忆单元、实体、关系、链接
- 本地文件存储 / 后续对象存储

这一层的目标是把“原著内容”和“业务数据”转成可被生成流程消费的上下文。

#### 第三层：Agent 增强层

负责多步推理、工具调用、自校验和长期辅助创作。

- 角色提取 Agent
- 世界观一致性 Agent
- 角色一致性 Agent
- 大纲规划 Agent
- 伏笔追踪 Agent
- 风格 Reviewer / 改写 Agent

这一层不替代主生成链路，而是在主链路前后进行增强、校验或辅助规划。

### 3.2 菜单能力地图

为了避免“全菜单 Agent 化”停留在概念层，后续能力需要按菜单挂接到统一的资产工作台思路下。

| 菜单 | 当前已具备 | 下一步 Agent 能力 | 主要依赖阶段 |
|---|---|---|---|
| 概览 | 项目基础信息展示 | 项目摘要、能力缺口分析、下一步建议 | D、E |
| 章节 | 章节结构与生成链路基础 | 大纲规划、局部重生成、章节树导入 | C、D |
| 角色 | 角色实体与提取样板 | 角色提取、角色归并、角色一致性守护 | D |
| 世界观 | 世界观实体已存在 | 设定提取、规则补全、冲突检查 | D、E |
| 文风 | 风格资产已存在 | 文风画像、风格守护、改写建议 | D、E |
| 草稿生成 | 现有 Skill 主链路 | 规划-生成-审校链式 Agent、生成后守护 | C、D |
| 原著导入 | 导入、切片、向量化、进度展示 | 导入后自动提取候选资产、任务中心入口 | B、D |

这张能力图的目的不是把每个菜单都做成一个聊天框，而是让每个菜单都成为“项目资产工作台”，统一复用 Agent 运行时、工具系统、建议审核层与任务中心。

## 4.1 阶段 A：最小可运行底座

### 目标

在不破坏现有 MuseSpace 业务结构的前提下，补齐记忆层运行所需最小基础设施。

### 范围

- 确定开发环境方案
- PostgreSQL 16 + pgvector 启用
- Schema 分层
- 本地文件存储
- 记忆层实体与 DbContext 接入
- Migration 规范统一

### 重点设计

- 存储以 PostgreSQL 为中心，前期不引入额外中间件作为前置依赖
- 文件落本地 `dev-data/files/`，数据库仅保存逻辑 Key
- 记忆层表与业务表逻辑分层，避免后续扩展时互相污染

### 本阶段交付

- 开发环境可复用
- 数据库扩展和 schema 可用
- 本地文件存储接口可用
- 项目可编译运行，现有 Skill 链路不受影响

## 4.2 阶段 B：原著导入与基础记忆流水线

### 目标

打通“导入 TXT / Markdown 小说 → 切片 → 向量化 → 检索”的初级记忆闭环。

### 核心流程

```text
前端上传文件
→ NovelsController 接收并去重
→ IStorageService 保存原文
→ Hangfire 投递 ChunkNovelJob
→ 切片结果写入 novel_chunks
→ ContinueJobWith 触发 EmbedNovelJob
→ 写入 memory.chunk_embeddings
→ SignalR 推送进度
→ 前端展示处理状态
```

### 重点模块

#### 文件接收与去重

- API 接收 `IFormFile`
- 计算 SHA256 去重
- 校验 Content-Type 与文件内容
- 创建 `novels` 记录并保存原文件

#### 文本切片

- 初期采用纯 C# 规则切片
- 优先按自然段和句号边界断开
- 控制 chunk 大小与 overlap，避免上下文断裂

#### 后台任务调度

- 采用 Hangfire 串联切片任务与向量化任务
- 支持任务重试、状态可见、进程重启恢复

#### Embedding 与向量存储

- 抽象 `IEmbeddingClient`
- 统一记录 `ModelName` 与向量维度
- 使用 pgvector 建立 HNSW 索引

#### 实时进度反馈

- SignalR 主推送
- 状态查询接口作为 fallback

### 本阶段交付

- 支持 TXT / Markdown 导入
- 完成切片与向量化
- 检索服务可返回相关原著段落
- 前端可看到导入和处理进度

## 4.3 阶段 C：记忆接入生成主链路

### 目标

让原著检索结果真正参与续写与番外生成，而不是停留在“可查到”。

### 核心思路

在 `StoryContextBuilder` 中增加记忆检索环节，将“原著相关片段”与已有结构化业务数据一起注入 Prompt。

### 注入结构

生成上下文统一拆成两类：

- 结构化上下文：`Character`、`WorldRule`、`StyleProfile`、`Chapter`
- 文本记忆上下文：原著检索结果、后续 memory items、实体摘要

### 推荐做法

```text
生成前输入目标
→ 生成查询 embedding
→ 检索 topK 原著 chunk
→ 与 Character / WorldRule / Chapter 等上下文一起组装
→ 填充 Prompt 模板中的 novel_context 区块
→ 调用现有 Skill 生成
```

### 检索策略演进

第一步：纯向量检索，快速跑通

第二步：引入关键词检索 + RRF 融合，形成 Hybrid RAG

第三步：结合来源、重要度、访问次数、时间衰减做排序优化

### 本阶段交付

- 原著检索结果可稳定进入生成 Prompt
- 续写/番外质量出现可感知提升
- 形成结构化数据 + 原著文本的双源融合模式

### 本阶段对大纲能力的意义

阶段 C 不直接承载完整的大纲 Agent 落地，但它决定后续大纲规划是否可靠。其核心价值在于：

- 为大纲规划提供稳定的原著检索输入，而不是凭空规划。
- 把角色、世界观、章节、文风等约束整理成可消费的上下文基础。
- 为后续“续写大纲”“番外大纲”“局部剧情规划”提供可信的数据来源。

---

## 4.4 阶段 D：Agent 化增强接入

### 目标

在不替代 Skill 主链路的前提下，引入可复用的 Agent 运行时，让多步任务、工具调用、自校验能力逐步落地。

### 统一定位

- `ISkill` 负责单步生成
- `IAgentRunner` 负责多步决策与工具执行
- 生成主流程仍以 Skill 为中心，Agent 作为前置辅助或后置校验

### 优先落地顺序

#### D1：P0 基础设施归一化

- 固化 `IAgentRunner`、`IAgentTool`、`AgentDefinition`
- 沉淀 `AgentRun` 持久化、日志埋点、Feature Flag
- 将现有角色卡提取重构为标准 Agent

#### D2：一致性守护

- 世界观一致性 Agent
- 角色一致性 Agent

它们作为草稿生成后的异步守护能力运行，失败不阻塞主流程。

#### D3：创作起点辅助

- 大纲规划 Agent
- 章节结构生成与导入

#### D3.5：菜单 Agent 化与统一交互

- 原著导入后的自动提取流水线
- 统一任务中心
- 统一建议审核层
- 概览 / 章节 / 角色 / 世界观 / 文风 / 草稿生成的统一 Agent 入口

#### D4：长期创作辅助

- 伏笔追踪 Agent
- 风格 Reviewer
- 改写 Agent 工具组

### Agent 与记忆层的结合方式

Agent 不应直接绕开现有数据层，而应以工具方式访问记忆与业务资产，例如：

- `GetCharacterCardTool`
- `GetWorldRulesTool`
- `ListChaptersTool`
- `SearchNovelChunksTool`
- `ListOpenThreadsTool`

这样可以保证 Agent 行为可控、可审计、可替换。

### 本阶段交付

- Agent 运行时标准化
- 至少 1 个现有 Agent 迁移完成
- 一致性守护能力可异步运行
- 大纲规划能力可输出结构化候选结果并导入章节树
- 统一任务中心与建议审核层具备雏形
- Agent 运行轨迹可观测、可灰度、可关闭

---

## 4.5 阶段 E：记忆网络深化与工业化扩展

### 目标

在主链路和 Agent 增强都跑通后，把记忆从“原著 chunk 检索”升级到“可组织、可关联、可长期积累”的体系。

### 重点能力

#### 记忆单元层

- `memory_items`
- `memory_embeddings`
- `memory_links`

用于承载摘要、重要片段、短期记忆、长期记忆及其关系。

#### 实体关系层

- `entities`
- `entity_embeddings`
- `entity_relations`

先使用 PostgreSQL 实现默认关系层，后续当复杂度达到阈值时再评估 Neo4j / GraphRAG。

#### 线索与伏笔层

- `PlotThread`
- 埋点、提醒、回收、状态追踪

#### 规划与建议层

- `OutlinePlan` / `OutlineVolume` / `OutlineChapter`
- `AgentSuggestion` / 审核状态 / 版本化结果

用于承载大纲、候选资产、待确认建议和局部重生成结果。

#### 工业化基础设施

- Redis 用于缓存、限流、任务协同
- 对象存储替代本地文件
- 独立 Worker 承接重型异步任务
- 更完整的管理后台与评估基建

### 引入时机建议

只有在以下情况明显出现时，才建议进入工业化扩展：

- 导入与向量任务耗时明显拖慢 API 进程
- 记忆数据量开始逼近单库单表可接受边界
- Agent 数量变多，成本与可观测需求显著提高
- 需要跨项目、跨用户、跨环境做更强隔离和审计

在这些条件没有明显成立前，默认仍应坚持：

- 先用 PostgreSQL 落地关系层与长期记忆层。
- 先用本地存储 + `IStorageService` 抽象承接文件。
- 先用 API 进程 + Hangfire 跑通 Agent 与提取流程。
- 先用单 Agent + 工具链完成菜单 Agent 化，再评估 Multi-Agent。

## 7. 技术引入判定表

为了避免在实现过程中反复争论“要不要现在上某个技术”，这里统一定义引入边界。

| 技术 / 能力 | 解决问题 | 最早建议阶段 | 明确引入条件 |
|---|---|---|---|
| `memory_items` | 让原著片段、摘要、候选知识可长期组织 | E | 原文 chunk 已明显过多，单靠 chunk 回填 Prompt 成本过高 |
| `entities` / `entity_relations` | 支撑角色、组织、地点、规则的结构化关系 | E 前段 | 角色关系、设定关系需要跨章节和跨资产复用 |
| `PlotThread` | 支撑伏笔、回收、主支线协同 | D 后段 / E 前段 | 大纲规划与长篇续写需要持续维护线索生命周期 |
| 对象存储 | 替代本地文件，提升协作与部署稳定性 | E | 多机部署、多人协作、文件规模增长到本地存储不再方便 |
| Redis | 缓存、限流、短期状态、任务协同 | E | Agent 任务量和热点读取明显上升，数据库直读成本过高 |
| 独立 Worker | 拆出重型异步任务，减轻 API 压力 | E | embedding、批量提取、摘要、Agent 任务开始拖慢 API 进程 |
| Neo4j | 复杂关系图谱与多跳查询 | E 中后段 | PostgreSQL 关系表已难以低成本支撑多跳关系查询 |
| GraphRAG | 图 + 向量 + 关键词的复合检索 | E 中后段 | 已有实体关系层和评估样例，且普通检索已无法满足质量目标 |
| Multi-Agent | 复杂任务拆分协同 | E 后段 | 单 Agent + 工具链已经无法稳定承载复杂规划与守护流程 |

## 8. 建议的实际开发顺序

综合可落地性、依赖关系与业务收益，建议按如下顺序推进：

1. 阶段A收口：数据库、扩展、schema、本地文件存储与 Migration 规范补齐
2. 阶段B回归：`Novel` 导入接口、切片、Hangfire、Embedding、检索链路做完整回归和补边界
3. 阶段C优化：`StoryContextBuilder`、Prompt 融合、上下文预算与评估样例补齐
4. 阶段D主线开发：先落统一任务中心、建议审核层和大纲规划 Agent
5. 阶段D扩展：再推进一致性守护、文风 Reviewer 与菜单 Agent 化
6. 阶段D长期增强：最后进入伏笔追踪、PlotThread 与局部重生成
7. 阶段E预研：只有在负载、复杂度、评估结果都支持时，才进入 GraphRAG、Neo4j、Redis、独立 Worker 等扩展

这个顺序的核心理由是：先让记忆“可导入、可检索、可注入”，再让 Agent “可调度、可校验、可规划”，最后再让系统“可长期协作、可工业化扩展”。

## 10. 风险与控制策略

### 10.1 范围蔓延风险

风险：前期过早引入太多中间件和高级框架，导致迟迟跑不通主线。

控制：所有新增能力必须回答“是否直接改善原著导入和续写体验”，否则后置。

### 10.2 生成链路复杂度上升风险

风险：记忆检索、Agent 校验加入后，链路过长且难排查。

控制：主生成链路始终保持单独可运行，增强能力默认可降级关闭。

### 10.3 成本与性能风险

风险：Embedding、Agent、多轮调用会带来成本与吞吐压力。

控制：

- Embedding 任务后台异步化
- Agent 成本预算与限流
- 先使用便宜模型，必要时再允许用户升级

### 10.4 数据模型过早定型风险

风险：在使用场景未稳定时一次性铺太多表，后续难以调整。

控制：先落最小可用模型，实体关系和高级记忆表按阶段引入。

### 10.5 菜单 Agent 化碎片化风险

风险：每个菜单各自长出一套 Agent 逻辑、提示词和状态流转，导致系统越来越散。

控制：统一通过 Agent 运行时、统一任务中心、统一建议审核层和统一工具链接入，不允许页面各自重复造轮子。

---

## 11. 最终落地建议

如果只看当前一到两个迭代，最值得优先完成的不是“完整记忆系统”或“复杂 Agent 平台”，而是下面这条短链路：

```text
原著上传
→ 切片
→ 向量化
→ 提取候选角色 / 世界观 / 文风
→ 注入现有 SceneDraftSkill
→ 大纲规划与章节树生成
→ 生成后增加一致性守护 Agent
```

这条链路一旦稳定，MuseSpace 就已经具备了区别于普通写作工具的核心能力：

- 能理解并复用原著内容
- 能结合项目内角色、世界观、风格进行创作
- 能根据导入原著与项目资产规划大纲和章节骨架
- 能在生成后给出一致性检查与修正建议

后续所有高级能力，包括记忆网络、PlotThread、Reviewer、GraphRAG、Multi-Agent，本质上都应该建立在这条主链路稳定的前提之上。

---

## 12. 结论

三份原始计划并不冲突，它们分别覆盖了不同层面的建设重点：

- 记忆层精简计划解决“先把记忆底座和阶段节奏定下来”
- 初级记忆方案解决“如何把导入、切片、向量化、检索真正做出来”
- Agent 过渡计划解决“如何把已有数据资产转化为更智能的辅助能力”

整合后的统一实施思路应当是：

**以现有 Skill 主链路为稳定核心，以基础记忆流水线为第一优先级，以“菜单 Agent 化 + 大纲规划 + 建议审核”为第二阶段价值放大器，以更深层记忆网络和工业化能力作为后续扩展。**

这也是当前 MuseSpace 最稳妥、最务实、同时对产品价值提升最直接的推进路径。
