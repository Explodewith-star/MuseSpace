# C-4 草稿生成评估样例集

> 阶段C任务清单 C-4 的产出。提供一组**固定**的草稿生成（`SceneDraftSkill`）回归样例，作为后续检索策略调整（Hybrid RAG / 重排序 / 长度预算调整）的人工评估基线。
>
> **使用方式**：每次修改 `StoryContextBuilder` / Prompt 模板 / 检索常量后，挑取下表样例在两个分支或两次运行下分别跑一次草稿生成，记录差异，**不要靠主观感觉判断**。

---

## 1. 评估目标

每条样例围绕**三个维度**做人工对比：

1. **角色一致性**（C-Cons）：输出是否与项目内 `Character` 卡的设定（性格 / 动机 / 说话方式 / 禁止行为）一致。
2. **世界观贴近度**（W-Cons）：输出是否引用或遵守 `WorldRule`（特别是 `IsHardConstraint = true` 的条目）。
3. **原著语境贴近度**（N-Cons）：输出是否复用了 `NovelContextSnippets` 中的设定/术语/语气，而非自由发挥。

打分采用三档：`✅ 通过` / `⚠️ 部分通过` / `❌ 不通过`，并在备注栏写出具体证据片段。

---

## 2. 测试项目准备（一次性）

为减少环境噪声，建议固定一个**评估专用项目**，包含：

- 至少 5 个角色（含 1 个有 `ForbiddenBehaviors` 字段的）
- 至少 6 条世界观规则（含 2 条 `IsHardConstraint = true`）
- 至少 1 本原著（已 Indexed）
- 至少 3 章已写定稿

> 评估时**不要**用真实创作项目，避免污染创作数据。

---

## 3. 固定样例集（V1，10 条）

| # | 场景类型 | SceneGoal | Conflict | 主要考察 | 难点 |
|---|---|---|---|---|---|
| S1 | 普通续写 | 主角与挚友在旧地重逢，平静中伏隐线 | 旧友的隐瞒令主角疑心 | C-Cons + N-Cons | 角色说话方式是否与原著一致 |
| S2 | 战斗场景 | 主角在第三幕高潮使用核心技能击退反派 | 体力濒临耗尽 vs 守护伙伴的执念 | W-Cons | 是否违反硬约束（如"力量上限"） |
| S3 | 情感独白 | 配角回忆童年丧亲，独自面对夜空 | 内心愧疚 vs 已成既定事实 | C-Cons | 是否符合该角色 `PersonalitySummary` |
| S4 | 设定密集 | 主角第一次进入"灵气浓度断层带"，向新读者解释规则 | 新人疑惑 vs 老规则不能破 | W-Cons | 是否准确复述规则措辞 |
| S5 | 番外（无主线推进） | 配角A与配角B日常对话，提到一名未登场角色 | 笑闹中略带疏离 | C-Cons | 是否乱编未登场角色设定 |
| S6 | 角色禁止行为 | 主角面临背叛诱惑，需要拒绝 | 被背叛的诱惑 vs 内心准则 | C-Cons | 是否触犯 `ForbiddenBehaviors` |
| S7 | 多角色场景 | 五人小队会议讨论下一步行动 | 路线分歧 | C-Cons | 各角色发言是否各自符合卡片 |
| S8 | 风格贴近度 | 描写一场雨夜街头追逐 | 节奏紧张 | N-Cons | 句式 / 描写密度是否匹配 `StyleProfile` |
| S9 | 检索空结果回退 | 描写一段在原著中**不存在**的设定（虚构科技元素） | 调试空召回路径 | 主链路稳定性 | 应能正常生成而不报错 |
| S10 | 长上下文压力 | 在已有 50 章定稿的项目里续写第 51 章 | 检验上下文裁剪 | 预算治理 | 不超出 token 上限 |

> S9 / S10 主要用于**回归性测试**，验证 C-1（空结果回退）和 C-3（预算控制）逻辑。

---

## 4. 评估记录模板

每次跑评估时，复制下表填写：

```
日期：YYYY-MM-DD
分支 / 配置：例如 main / NovelTopK=5,Threshold=0.3,Budget=3000
评估人：

| # | C-Cons | W-Cons | N-Cons | 备注（证据片段或问题） |
|---|--------|--------|--------|------------------------|
| S1 | ✅ | ✅ | ⚠️ | 主角的口头禅未复用 |
| S2 | ✅ | ❌ | ✅ | 力量上限被打破：原文"...核心碎裂..." |
| S3 | ⚠️ | ✅ | ✅ | 性格符合，但描写过于现代 |
| ... |
```

历史评估结果建议另存到 `docs/evaluation/runs/YYYY-MM-DD-<分支>.md`。

---

## 5. 与 Hybrid RAG 引入的协同

引入 Hybrid RAG（关键词检索 + RRF 融合）时，**先跑一次本评估集作为 baseline**；切换为 Hybrid 后再跑一次，按以下规则判断是否上线：

- N-Cons 维度 ✅ 数量提升 ≥ 30%（专名 / 术语召回主要受益区）
- C-Cons / W-Cons 维度不应回退
- 不出现新增的 ❌（不能让 Hybrid 引入不相关片段）

只有满足以上条件，Hybrid 才能切为默认检索策略。

---

## 6. 当前 V1 限制

- 评估为人工而非自动评分；阶段 E 末段考虑接入自动 Judge LLM。
- 样例集偏向"修真 / 都市玄幻"题材，后续如有其他题材项目入库，应扩充对应专题样例。
- 暂不评估生成的多样性、剪辑率等指标。

---

## 7. 维护约定

- 每条样例的 `SceneGoal` / `Conflict` 文案一旦定稿，**不得随便修改**，否则历史结果失去可比性。
- 新增样例追加到 V2 节，保留 V1 编号不变。
- 评估专用项目的 ID 写入 `appsettings.Development.json` 的可选键 `Evaluation:ProjectId`，便于一键切换。
