## 总体流程

```
前端上传文件
    │
    ▼
NovelsController
POST /api/projects/{projectId}/novels
    │ 校验文件类型 + SHA256 去重
    │ 写 novels 表 (status=Pending)
    │ 保存文件到 dev-data/files/raw/
    │
    └─ Hangfire.Enqueue<ChunkNovelJob>(novelId)
                │
                ▼ Job 1：切片
         ChunkNovelJob
         读文件 → 文本切片 → Npgsql COPY 批量写 novel_chunks
         更新 novels.status = Indexed
                │
                └─ Hangfire.ContinueJobWith<EmbedNovelJob>
                              │
                              ▼ Job 2：向量化
                        EmbedNovelJob
                        分批调用 IEmbeddingClient
                        → 写 memory.chunk_embeddings
                        → SignalR 推送进度
                        → 全部完成推送 Done
                              │
                              ▼
                    前端 SignalR 实时更新状态 UI
```

## 模块一：文件接收与去重

**技术：** ASP.NET Core `IFormFile` + `SHA256` + `IStorageService`

### API 设计

```
POST /api/projects/{projectId}/novels
  multipart/form-data 接收文件
  → 读取字节流，计算 SHA256
  → 查 novels 表是否同 hash + projectId 已存在（去重）
  → 写文件到 dev-data/files/raw/{novelId}.txt（通过 IStorageService）
  → 插入 novels 记录（status = Pending）
  → Hangfire.Enqueue<ChunkNovelJob>(novelId)
  → 返回 { novelId, status: "pending" }

GET /api/projects/{projectId}/novels/{novelId}/status
  → 查 novels.status + novel_chunks 总数（轮询 fallback）

GET /api/projects/{projectId}/novels
  → 列出该项目所有已导入小说
```

### 关键细节

- 文件逻辑 Key 格式：`raw/{novelId}.txt`，不存绝对路径
- `IStorageService` 接口抽象文件读写，底层目前是本地磁盘，后续可替换为 COS/MinIO
- 文件大小限制：API 层 `options.Limits.MaxRequestBodySize` + nginx `client_max_body_size` 双层配置
- 文件类型校验：Content-Type 为 `text/plain` 或 `text/markdown`，魔数字节校验

## 模块二：后台任务调度

**技术：** `Hangfire.AspNetCore` + `Hangfire.PostgreSql`

### 包引入位置

| 包 | 引入位置 |
|---|---|
| `Hangfire.AspNetCore` | `MuseSpace.Api` |
| `Hangfire.PostgreSql` | `MuseSpace.Infrastructure` |

### 核心价值

| 对比点 | Channels（轻量方案）| Hangfire（选用方案）|
|---|---|---|
| 进程重启任务丢失 | 会丢失 | 持久化到 PG，重启自动恢复 |
| 任务状态可视化 | 无 | `/hangfire` Dashboard 一目了然 |
| 失败自动重试 | 需自己写 | 内置，指数退避，默认 10 次 |
| 任务链（切片→向量化）| 需手动衔接 | `ContinueJobWith` 原生支持 |

### 注册配置（Program.cs）

```csharp

    {

app.MapHangfireDashboard("/hangfire", new DashboardOptions
{
    Authorization = app.Environment.IsDevelopment()
        ? [new AllowAllDashboardAuthorizationFilter()]
```

## 模块三：文本切片

**技术：** 纯 C# 字符串处理，封装为 `NovelTextChunker`（无额外依赖）

### 切片策略

| 参数 | 值 | 说明 |
|---|---|---|
| 目标大小 | 800 字符/片 | 覆盖约 400～600 汉字，语义足够完整 |
| 重叠 | 80 字符 | 保留跨切片边界的上下文衔接 |
| 优先断点 | `\n\n` > `。！？` > 强制截断 | 尽量在自然段落或句子边界切片 |

### 切片量估算

```
85 万字 ÷ 800 字/片 ≈ 1062 个切片
```

### 接口设计

```csharp
public interface INovelTextChunker
{
    IReadOnlyList<NovelChunkCandidate> Split(string content, Guid novelId, Guid projectId);
}

public record NovelChunkCandidate(
    Guid Id, Guid NovelId, Guid StoryProjectId,
    int ChunkIndex, string Content,
    int CharCount, int StartOffset, int EndOffset
);
```

## 模块四：切片批量写入

**技术：** `Npgsql.BeginBinaryImportAsync`（PostgreSQL COPY 二进制协议）

### 为什么不用 EF Core SaveChanges

| 方案 | 1000 条写入耗时 | 备注 |
|---|---|---|
| EF Core `AddRange` + `SaveChanges` | ~1000ms | 1000 次往返 INSERT |
| EF Core `BulkInsert`（第三方扩展）| ~200ms | 有额外依赖 |
| **Npgsql COPY BINARY** | **~20ms** | PostgreSQL 原生批量协议，50 倍性能优势 |

### 实现骨架

```csharp
public async Task ExecuteAsync(Guid novelId, PerformContext? context)
{
    // 1. 从数据库加载 Novel 记录，读取文件
    // 2. 调用 INovelTextChunker.Split 得到切片列表
    // 3. Npgsql COPY BINARY 写入 novel_chunks
    await using var writer = await conn.BeginBinaryImportAsync(
        "COPY novel_chunks (id, novel_id, story_project_id, chunk_index, " +
        "content, char_count, start_offset, end_offset, is_embedded, created_at) " +
        "FROM STDIN (FORMAT BINARY)");

    foreach (var c in chunks)
    {
        await writer.StartRowAsync();
        await writer.WriteAsync(c.Id,             NpgsqlDbType.Uuid);
        await writer.WriteAsync(c.NovelId,        NpgsqlDbType.Uuid);
        await writer.WriteAsync(c.StoryProjectId, NpgsqlDbType.Uuid);
        await writer.WriteAsync(c.ChunkIndex,     NpgsqlDbType.Integer);
        await writer.WriteAsync(c.Content,        NpgsqlDbType.Text);
        await writer.WriteAsync(c.CharCount,      NpgsqlDbType.Integer);
        await writer.WriteAsync(c.StartOffset,    NpgsqlDbType.Integer);
        await writer.WriteAsync(c.EndOffset,      NpgsqlDbType.Integer);
        await writer.WriteAsync(false,            NpgsqlDbType.Boolean);
        await writer.WriteAsync(DateTime.UtcNow,  NpgsqlDbType.TimestampTz);
    }
    await writer.CompleteAsync();

    // 4. 更新 novels.status = Indexed
}
```

## 模块五：向量化（Embedding）

**技术：** 新增 `IEmbeddingClient` 接口，与 `ILlmClient` 同级，独立实现

### 接口定义

```csharp
// MuseSpace.Application/Abstractions/Llm/IEmbeddingClient.cs
public interface IEmbeddingClient
{
    Task<float[]> EmbedAsync(string text, CancellationToken ct = default);
    string ModelName { get; }
    int Dimensions { get; }
}
```

### 实现选型

| 实现 | 模型 | 维度 | 说明 |
|---|---|---|---|
| `OpenRouterEmbeddingClient` | text-embedding-3-small | 1536 | 默认实现，复用现有 HttpClient |
| `DeepSeekEmbeddingClient` | deepseek-embed | 1024 | 已建 vector(1024)，成本更低 |

> 当前 Migration 已建 `vector(1024)`，默认使用 DeepSeek Embedding 端点。

### EmbedNovelJob 处理逻辑

```
取出所有未 embedding 的 novel_chunks（is_embedded = false）
  │
  每批 20 个并发调用 IEmbeddingClient.EmbedAsync
  │
  写入 memory.chunk_embeddings（vector + model_name）
  │
  更新 novel_chunks.is_embedded = true
  │
  SignalR 推送：{ novelId, stage: "embedding", done: i, total: n }
  │
  全部完成 → SignalR 推送 ImportDone
```

## 模块六：向量存储与检索

**技术：** pgvector + `memory.chunk_embeddings` 表 + 原生 SQL `FromSqlRaw`

### 表结构（已在 InitialCreate Migration 中建立）

```sql
CREATE TABLE memory.chunk_embeddings (
    id               UUID PRIMARY KEY,
    chunk_id         UUID NOT NULL REFERENCES novel_chunks(id),
    story_project_id UUID NOT NULL,
    model_name       VARCHAR(200) NOT NULL,
    embedding        vector(1024) NOT NULL,
    created_at       TIMESTAMPTZ
);

-- HNSW 索引（首次运行后手动建立，通过 Migration）
CREATE INDEX idx_chunk_embeddings_hnsw
    ON memory.chunk_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

### 检索服务

```csharp
public interface INovelMemorySearchService
{
    Task<IReadOnlyList<NovelChunkSearchResult>> SearchAsync(
        Guid projectId, string queryText, int topK = 5);
}
```

检索实现使用 `<=>` 余弦距离：

```csharp
var results = await _db.ChunkEmbeddings
    .FromSqlRaw("""
        SELECT ce.*, nc.content AS chunk_content
        FROM memory.chunk_embeddings ce
        JOIN novel_chunks nc ON nc.id = ce.chunk_id
        WHERE ce.story_project_id = {0}
          AND ce.model_name = {1}
        ORDER BY ce.embedding <=> {2}::vector
        LIMIT {3}
        """, projectId, modelName, pgVector, topK)
    .ToListAsync();
```

## 模块七：实时进度推送

**技术：** `Microsoft.AspNetCore.SignalR`（ASP.NET Core 内置，无需额外引入）

### Hub 设计

```csharp
// MuseSpace.Api/Hubs/NovelImportHub.cs
public class NovelImportHub : Hub
{
    // 客户端连接后调用：await connection.invoke("JoinNovelGroup", novelId)
    public async Task JoinNovelGroup(string novelId)
        => await Groups.AddToGroupAsync(Context.ConnectionId, novelId);

    public async Task LeaveNovelGroup(string novelId)
        => await Groups.RemoveFromGroupAsync(Context.ConnectionId, novelId);
}
```

### 推送事件定义

| 事件名 | 推送时机 | 数据结构 |
|---|---|---|
| `ChunkProgress` | 切片阶段完成（一次性）| `{ novelId, stage: "chunking", done, total }` |
| `EmbedProgress` | 每批 embedding 完成 | `{ novelId, stage: "embedding", done, total }` |
| `ImportDone` | 全部完成 | `{ novelId, status: "success" }` |
| `ImportFailed` | 任意阶段失败 | `{ novelId, error: "..." }` |

### Job 内推送方式

```csharp
// Job 构造注入 IHubContext<NovelImportHub>
await _hubContext.Clients.Group(novelId.ToString())
    .SendAsync("EmbedProgress", new { novelId, done = i, total });
```

> 前端始终保留 `GET /api/projects/{id}/novels/{novelId}/status` 作为轮询 fallback。
