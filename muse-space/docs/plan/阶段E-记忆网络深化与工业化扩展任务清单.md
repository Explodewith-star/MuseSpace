# 阶段E：记忆网络深化与工业化扩展任务清单

> 阶段目标：在主生成链路、基础记忆、Agent 增强都稳定后，再把系统升级为更完整的记忆网络和工业化运行体系。
>
> 当前状态：尚未开始。本阶段不是当前最近一轮开发主线。

---

## 1. 阶段范围

- 记忆单元与摘要化记忆
- 实体关系网络
- 规划层与建议层的长期化沉淀
- PlotThread 长期线索追踪
- Redis / 对象存储 / 独立 Worker
- 更深层的 GraphRAG 与高级检索
- Agent 成本控制和评估基建

---

## 2. 可执行任务清单

### E-1 记忆单元层建设

目标：从“原著 chunk 召回”升级到“可整理、可摘要、可长期访问”的记忆层。

任务：

- 建 `memory_items`
- 建 `memory_embeddings`
- 建 `memory_links`
- 定义记忆来源、重要度、访问次数、过期策略
- 明确哪些 chunk 应被提升为高价值记忆项
- 让已确认的大纲、角色建议、世界观建议也能沉淀为长期可检索记忆

实现思路：

- 初期不要对所有 chunk 自动摘要
- 先从高价值片段、角色相关描述、设定相关描述开始沉淀 memory item
- 长期记忆层不只服务原著检索，也要服务大纲规划、线索追踪和后续 Agent 守护

验收：

- 能从原著和业务资产中抽取出可单独检索的记忆单元

状态建议：未开始。

### E-2 实体关系层建设

目标：让角色、组织、地点、规则、事件之间的关系可被查询和推理。

任务：

- 建 `entities`、`entity_embeddings`、`entity_relations`
- 明确实体抽取入口是人工录入、规则抽取还是 Agent 辅助抽取
- 建立实体与 chunk、章节、角色卡之间的映射关系
- 为后续 GraphRAG 保留升级路径
- 保留 `IEntityGraphService` 或等效抽象，确保 PostgreSQL 与 Neo4j 可平滑切换

实现思路：

- 第一版仍优先使用 PostgreSQL 关系表
- 只有当关系查询复杂度、图遍历需求、维护成本明显上升时，才考虑 Neo4j

验收：

- 能围绕角色或规则查询其关键关系和来源片段

状态建议：未开始。

### E-3 PlotThread 与长期线索追踪

目标：支撑长篇小说的伏笔、回收、悬念链路管理。

任务：

- 建 `PlotThread` 表和状态模型
- 把章节、场景、角色、事件和线索关联起来
- 允许 Agent 提示新伏笔、提醒未回收线索、推荐关闭时机
- 建立线索板 UI

实现思路：

- PlotThread 是创作协同数据层，不应只存在于 Agent 输出里
- 先把数据模型做实，再让 Agent 读写它

补充要求：

- `OutlinePlan` / `OutlineChapter` 应与 `PlotThread` 建立映射关系，支持从大纲阶段提出线索，到正文阶段落地与回收的持续追踪。
- 线索层必须能同时承接“大纲阶段提出的线索”和“正文阶段实际出现的线索”，避免两套系统并行。

验收：
- 能完整追踪一条伏笔从埋设到回收的生命周期

状态建议：未开始。

### E-3.5 规划层与建议层长期化

目标：让阶段D形成的大纲、建议、审核结果成为可追踪、可版本化、可复用的长期资产。

任务：

- 为 `OutlinePlan`、`OutlineVolume`、`OutlineChapter` 增加版本和来源追踪能力。
- 为 `AgentSuggestion` 增加历史记录、批量操作记录和回溯能力。
- 让已应用的大纲和建议可进入长期记忆或评估集，用于后续 Agent 优化。

实现思路：

- 阶段D解决“能用”，阶段E解决“长期可演进”。
- 如果没有版本化和来源追踪，后续大纲重生成、线索回看和质量评估会越来越困难。

验收：

- 任意一份大纲或建议都可以回溯来源、版本和最终应用结果。

### E-4 工业化基础设施升级

目标：解决数据量、任务量、协作方式扩大后的工程压力。

任务：

- Redis 用于缓存、限流、去重辅助或短期状态管理
- 对象存储替代本地文件存储
- 独立 Worker 承担 embedding、摘要抽取、批处理 Agent 任务
- 增强后台监控、告警和重试治理

技术判断重点：

- Redis 先服务缓存、任务协同和高频状态读取，不要先为了“架构完整”引入。
- 对象存储优先解决多人协作、部署和文件可靠性问题，不作为单机开发前置条件。
- Worker 优先承接 embedding、批量提取、大纲批处理、长期记忆整理等重任务。

实现思路：

- 工业化组件不是架构荣誉勋章，只在负载和协作需求明确出现时才引入
- 每个组件引入前都必须回答“当前痛点是什么，替换后可量化改善什么”

验收：

- API 进程从重型异步任务中解耦
- 文件和缓存体系适应多人协作与更大数据量

状态建议：未开始。

### E-5 检索与推理高级化

目标：在基础 RAG 和 Agent 都稳定后，再升级到更强的知识组织和推理能力。

任务：

- 引入重排序器或更细粒度召回打分
- 评估并试点 GraphRAG
- 让 Agent 能联合使用实体关系、memory items、PlotThread 和原著 chunk
- 构建评估集，比较普通检索、Hybrid、GraphRAG 的收益差异
- 让大纲规划、线索追踪、角色守护分别拥有可对比的评估样例，而不是只比较单一问答检索效果

实现思路：

- 高级检索必须以评估数据为前提，而不是靠感觉替换
- GraphRAG 应先做小范围试点，例如只用于复杂人物关系或大型世界观项目

验收：

- 能明确说出高级检索相对基础检索提升了什么、成本增加了什么

状态建议：未开始。

## 3. 实现顺序

1. 先做 memory items
2. 再做规划层与建议层的长期化
3. 再做 entities 与 entity_relations
4. 再做 PlotThread
5. 然后根据真实压力引入 Redis / 对象存储 / Worker
6. 最后再试点 GraphRAG 和更重的推理增强

---

## 4. 高级能力的真正引入时机

### Hybrid 作为默认检索

引入时机：阶段C后期到阶段E前期之间即可完成，不必拖到 GraphRAG。

### GraphRAG

引入时机：阶段E中后段。

适用前提：

- 已有实体层和关系层数据
- 已有清晰问题需要图遍历解决
- 有评估样例可证明比 Hybrid 更值

不满足这些前提时，不要引入。

### Neo4j

引入时机：仅当 PostgreSQL 关系表和普通索引无法继续低成本维护图关系查询时。

建议前提：

- 已通过 PostgreSQL 把 `entities` / `entity_relations` 跑通
- 已保留关系服务接口抽象，迁移成本可控
- 已出现明确多跳查询和图算法需求

### 独立 Worker

引入时机：当 embedding、摘要抽取、批量 Agent 任务显著拖慢 API 进程时。

### Multi-Agent

引入时机：当单 Agent + 工具已经明显无法承载复杂任务编排，且团队已具备充分可观测与调试手段时。

---

## 5. 阶段完成定义

- 记忆从 chunk 检索升级到结构化记忆网络
- 大纲、建议、线索等中间资产可长期维护和版本化
- 长篇线索和实体关系可持续维护
- 工业化组件只在真实需要时引入且收益明确
- 高级检索能力有评估闭环，不靠主观判断推进
