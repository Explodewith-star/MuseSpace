## 一、问题总览

### 根本缺陷

系统中所有内容（角色、世界观、伏笔、事实）都是"项目级平铺"，缺少**可见性作用域**。  
导致：角色污染、番外间矛盾、伏笔越界、首章越界——本质都是**同一个问题的不同表现**。

### 确认的问题清单

| # | 问题 | 根因 | 优先级 |
|---|------|------|:---:|
| A | 原创草稿使用了原著角色（杨间等） | Character 无 SourceNovelId | P0 |
| B | 草稿内容跨章节错位 | A的后果，角色注入错误导致 LLM 发挥 | P0 |
| C | 批次之间故事断裂感 | 新大纲无法引用前置批次内容 | P1 |
| D | 番外A和番外B角色设定矛盾 | 番外间无共享事实机制 | P2 |
| E | 世界观规则：原著规则污染原创模式 | WorldRule 无 SourceNovelId | P1 |
| F | 伏笔追踪：AI写第三部时看到番外伏笔 | PlotThread 无归属信息 | P2 |
| G | 一致性检查：拿原著角色验证原创内容 | ConsistencyCheck 角色池未过滤 | P2 |
| H | 建议中心：应用后来源丢失 | CharacterSuggestionApplier 未传递 SourceNovelId | P0 |
| I | 番外章节事件影响主线 Canon 读取 | ChapterEvent/CanonFact 无大纲归属 | P2 |

## 二、核心架构：作用域链模型

```
Project（项目）        ← 所有故事线都能看到
  └─ Chain（故事链）   ← 同一条主线或系列番外内可见
      └─ Outline（批次/单部番外）← 只有本批次可见
```

### 新增实体

**OutlineChain（故事链）**
```
OutlineChain
  Id: Guid
  StoryProjectId: Guid
  Name: string              // "正传主线" / "张三外传系列"
  Mode: GenerationMode      // 决定链属性
  DisplayOrder: int
```

**StoryOutline 扩展**
```
StoryOutline
  + ChainId: Guid?              // 属于哪条故事链
  + ChainIndex: int             // 链内第几批（1,2,3...）
  + PreviousOutlineId: Guid?    // 直接前驱批次
```

### 现有实体修改

**Character**
```
+ SourceNovelId: Guid?   // 从哪部原著提取（null = 原创）
```

**WorldRule**
```
+ SourceNovelId: Guid?
+ Scope: "Project" | "Chain" | "Outline"
+ ScopeChainId: Guid?
+ ScopeOutlineId: Guid?
```

**PlotThread**
```
+ OutlineId: Guid?         // 在哪个批次中埋下
+ ChainId: Guid?           // 属于哪条故事链
+ Visibility: "ThisOutline" | "Chain" | "Project"
+ ResolvedInOutlineId: Guid?
```

**CanonFact**
```
+ Scope: "Project" | "Chain" | "Outline"
+ ScopeChainId: Guid?
+ ScopeOutlineId: Guid?
+ PromotedFromOutlineId: Guid?
```

## 三、功能改造方案

### 角色系统

- 角色页分区：原创角色 / 原著人物
- StoryContextBuilder 按模式过滤：
  - 原创模式 → 仅注入 SourceNovelId IS NULL 的角色
  - 续写模式 → 注入原创 + 指定原著角色
  - 番外模式 → 注入原创角色 + 该番外快照状态

### 世界观规则

- 原著导入时标记 SourceNovelId
- 原创模式过滤（同角色逻辑）
- 番外可创建局部规则

### 伏笔追踪

- 埋伏笔时指定生效范围（仅本批次 / 整条主线 / 全项目）
- StoryContextBuilder 按范围过滤

### 建议中心 + 原著导入

- 导入时保留 SourceNovelId 到角色/规则实体
- 角色页、世界观页明确展示来源分区

### 番外事实一致性

- 番外间默认隔离
- 完成后有"事实提升"步骤：将局部事实提升为全局 Canon
- 写新番外时自动读取全局 Canon

## 四、前端改造

### 导航

- 移除侧边栏「大纲」菜单
- 「章节」改名「创作」
- 创作页右上角加「使用指南」帮助弹框

### 创作页结构

```
[创作]
├─ 顶部：Mode Tabs（原创主线 / 支线番外 / 原著续写 / 扩写改写）
├─ 批次子导航：[第一部✓] [第二部✓] [第三部●] [+新建]
├─ 工具栏：[AI继续规划] [批量生草稿] [搜索]
└─ 章节列表（带前端过滤）
```

### 章节详情页

- 草稿/定稿：固定 5 行高度，点击"展开全文"才显示完整内容
- 面包屑：大纲名 > 第N章
- 返回时路由带 ?outline=uuid 自动定位

### 新增待完善项：章节边界闭环

- 背景：当前章节草稿即使发生"一章写穿多章"，用户也只能靠人工阅读发现，缺少显式验收反馈与返工入口。
- 目标 1：在章节详情页提供"按当前计划重生成本章"入口；若边界验收失败，前端直接展示失败原因，而不是只提示通用生成失败。
- 目标 2：为已落库草稿提供一次显式"章节边界验收"能力，并支持对项目/大纲内草稿做批量清查，筛出不连贯章节。
- 预期收益：把章节计划 -> 草稿生成 -> 边界验收 -> 重生成 的闭环补齐，降低首章越界、跨章抢跑、后续章不连贯的人工排查成本。

## 六、关键设计决策（已确认）

| 决策 | 结论 |
|------|------|
| 主线新建批次时是否强制续接 | 原创模式强制引导续接，支线模式可选 |
| 番外间 Canon 可见性 | 默认隔离，完成后有明确"提升到全局"步骤 |
