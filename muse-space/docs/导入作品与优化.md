# 导入作品与优化

## 1. 背景

本轮改动聚焦的是 MuseSpace 中“原著导入”这一条完整链路。最初暴露出来的问题并不是单点故障，而是几个问题叠加后共同影响了体验和稳定性：

- 用户不知道上传文件的大小限制，也不知道限制在哪里。
- 较大的 TXT 小说在导入时偶发后端报错，错误难定位。
- 即使上传成功，用户也看不到切片、向量化阶段的真实进度，容易误以为系统卡住。
- 11MB 左右的小说导入明显偏慢，且慢的主要原因并不在 HTTP 上传本身，而在上传后的切片和向量化阶段。
- 前后端对导入状态的定义不一致，刷新页面后也无法稳定恢复导入中的实时状态。

这几个问题如果只逐个修补，很容易变成“某个点修好了，但整体体验还是差”。因此这次的处理方式不是只修一两个 bug，而是把导入链路从“能跑”升级为“可观察、可恢复、可定位、可扩展”。

## 2. 本次改动目标

本次改动的目标可以概括为四点：

- 保证原著导入链路稳定，不再因为 Unicode 边界问题导致导入失败。
- 让用户看到真实的上传进度和处理进度，而不是只看到“上传完了以后一直没反应”。
- 针对当前真实瓶颈做性能优化，优先减少切片数量、减少 Embedding 调用次数、减少数据库写入次数。
- 把状态、进度、错误信息沉淀到数据库中，使页面刷新、SignalR 断线、任务失败后都能恢复和定位。

## 3. 为什么这样改

### 3.1 先修根因，不做表面绕过

后端报错的根因不是“文件大”，而是切片边界可能把代理对字符拆开，最终在写库时触发编码异常。若只是粗暴忽略异常或简单缩小 chunk，会让问题变隐蔽，但不会真正消失。因此优先修正切片逻辑，让 chunk 本身永远不落在非法 Unicode 边界上。

### 3.2 把导入拆成两个用户可理解的阶段

上传一个作品实际上有两个完全不同的阶段：

- 浏览器到 API 的文件上传阶段。
- API 接收完成后，后台执行切片和向量化阶段。

如果只展示“上传完成”，用户会误以为导入也已经完成；如果只展示“处理中”，用户又看不到上传本身到底卡没卡住。因此前端必须把这两个阶段拆开显示。

### 3.3 性能优化要打在真实瓶颈上

11MB 小说导入慢，核心不是 HTTP 文件传输，而是导入后需要生成大量 chunk，并对每个 chunk 做向量化。真正有效的优化是：

- 减少 chunk 数量。
- 批量调用 Embedding 接口。
- 批量更新嵌入状态。
- 尽量减少数据库逐条写入的开销。

这比只在上传控制器里做一些表面微调更有价值。

### 3.4 状态必须持久化，而不是只靠内存事件

SignalR 很适合做实时通知，但它不能代替持久化状态。如果进度只存在于内存或事件流里，页面刷新、连接断开、后台任务重启之后，前端就会失去上下文。因此这次把进度、错误、开始结束时间落在 Novel 表中，SignalR 负责“实时推送”，数据库负责“状态真相”。

## 4. 总体改动思路

这次的整体思路可以概括为一句话：

先让导入状态在后端成为完整、可恢复的数据模型，再让前端围绕这个模型做上传体验和处理进度展示，最后针对切片与向量化两个重型环节做针对性优化。

具体上分成五个层面：

- 上传入口优化。
- 切片稳定性修复。
- Embedding 批处理优化。
- 状态与进度持久化。
- 前端实时进度联动与刷新恢复。

## 5. 主要改动内容

### 5.1 上传入口与去重

在 `MuseSpace.Api/Controllers/NovelsController.cs` 中，导入入口继续保持“请求只负责接收、落库、投递后台任务”的职责，不在 HTTP 请求内做同步切片和向量化。

这部分主要做了几件事：

- 明确上传大小限制为 50MB，通过 `RequestSizeLimit(50 * 1024 * 1024)` 控制。
- 仍然只允许 `.txt` 和 `.md` 文件。
- 上传后用 SHA-256 计算文件哈希，并通过 `GetByProjectAndHashAsync` 做项目内去重。
- 复用同一个 `MemoryStream` 进行哈希计算和存储，减少一次额外字节拷贝。
- 上传成功后只创建 `Novel` 记录并串联 Hangfire 任务，不在请求中做重计算。

这样做的好处是上传请求的职责单一、返回更快，也为后续进一步把文件存储切到对象存储留出了空间。

### 5.2 切片稳定性修复

导入过程中最关键的稳定性修复在 `MuseSpace.Infrastructure/Novel/NovelTextChunker.cs` 和 `MuseSpace.Infrastructure/Jobs/ChunkNovelJob.cs`。

核心改动如下：

- 切片参数从更小粒度调整到目标 1600 字、重叠 120 字，减少 chunk 总数。
- 切片仍优先按自然段或句号边界断开，尽量不在生硬位置截断。
- 增加 `AlignStartBoundary` 和 `AlignEndBoundary`，确保 chunk 起止位置不会拆开代理对字符。
- 增加 `TrimRange`，避免 chunk 前后残留无意义空白。
- 在切片写库前增加非法 Unicode 检测，若发现问题会记录 chunk 索引、原始偏移和内容预览，便于快速定位。
- `ChunkNovelJob` 使用 PostgreSQL COPY BINARY 批量写入 chunk，保留较高写入效率。
- 在重试时先删除旧的 chunk 和 chunk embedding，保证任务幂等，避免 Hangfire 重试导致脏数据或重复数据。

这一块的本质不是“把代码写复杂”，而是把以前容易隐性失败的边界条件显式化和可诊断化。

### 5.3 Embedding 批处理优化

性能优化的重点放在 Embedding 阶段。

主要改动包括：

- 在 `MuseSpace.Application/Abstractions/Llm/IEmbeddingClient.cs` 中增加 `EmbedBatchAsync`，让 Embedding 客户端具备批量接口。
- 在具体实现中支持一次请求提交多条文本，降低外部 Embedding 服务的调用次数。
- 在 `MuseSpace.Infrastructure/Jobs/EmbedNovelJob.cs` 中把处理方式改为分批执行，当前批大小为 50。
- 每批 Embedding 完成后，在一个事务内写入 `memory.chunk_embeddings`，再批量标记 chunk 已向量化。
- 用 `MarkEmbeddedBatchAsync` 替代逐条更新，减少数据库 round-trip。
- 每批完成后更新 `Novel.ProgressDone` 和 `Novel.ProgressTotal`，并推送 SignalR 进度事件。

这部分的收益非常直接：同样一个文件，chunk 数变少了，调用次数变少了，数据库写入次数也变少了，导入总耗时自然下降。

### 5.4 导入状态模型统一

为了把“上传后到底进行到哪一步”描述清楚，这次统一了导入状态模型。

在 `MuseSpace.Domain/Enums/NovelStatus.cs` 中，状态被统一为：

- `Pending`
- `Chunking`
- `Embedding`
- `Indexed`
- `Failed`

同时，在 `MuseSpace.Domain/Entities/Novel.cs` 中新增了以下字段：

- `ProgressDone`
- `ProgressTotal`
- `LastError`
- `StartedAt`
- `FinishedAt`

这些字段同时通过 `NovelResponse` 返回给前端，并写入数据库迁移 `20260424153000_AddNovelImportProgress`。

这样做之后，Novel 不再只是“文件元数据 + 最终状态”，而是完整承载了一次导入任务的生命周期信息。

### 5.5 SignalR 进度通知与前端联动

前端体验的核心改动在 `muse-space-web/src/api/novels.ts`、`muse-space-web/src/composables/useNovelImportProgress.ts`、`muse-space-web/src/views/projects/novels/index.vue`。

主要思路如下：

- 上传阶段通过 Axios 的 `onUploadProgress` 获取浏览器真实上传进度。
- 后台处理阶段通过 SignalR 订阅 `ChunkProgress`、`EmbedProgress`、`ImportDone`、`ImportFailed` 事件。
- `useNovelImportProgress` 统一管理连接、加入组、离组、自动重连后重新订阅。
- 导入页维护本地 `activeImportTasks`，把“上传中”和“后台处理中”统一呈现在一块任务区域里。
- 页面首次加载时，如果发现后端返回的小说状态仍处于 `Pending`、`Chunking`、`Embedding`，会自动恢复对应的导入任务并继续追踪。
- SignalR 之外仍保留状态接口轮询作为兜底，避免断线后页面完全失去反馈。

这样前端就不再是“传完文件就等结果”，而是可以持续告诉用户：

- 文件是否还在上传。
- 是否已经开始切片。
- 向量化做到了多少。
- 是否失败以及失败原因是什么。

### 5.6 错误可定位性增强

这次不仅要“少报错”，也要“报错时能迅速知道问题在哪”。因此在后端加入了几类可定位信息：

- 非法 Unicode 的 chunk 索引、原文偏移、内容预览。
- 任务失败时写回 `Novel.LastError`。
- 记录 `StartedAt` 和 `FinishedAt`，方便判断失败发生在切片阶段还是向量化阶段。
- 通过日志输出切片总数、处理耗时、Embedding 批次进度。

这些信息组合起来，能够较快回答三个核心问题：

- 错在上传前、切片时还是向量化时。
- 是某个具体 chunk 的数据问题，还是外部 Embedding 服务问题。
- 这次慢到底慢在 chunk 数量、批处理速度，还是数据库写入。

## 6. 本次数据库变更

本次为了支撑进度持久化，新增了一条迁移：

- `20260424153000_AddNovelImportProgress`

迁移内容是给 `novels` 表增加以下字段：

- `ProgressDone`
- `ProgressTotal`
- `LastError`
- `StartedAt`
- `FinishedAt`

该迁移已在当前开发数据库上执行完成。

## 7. 已完成验证

本次改动完成后，已经做过以下验证：

- 后端 `MuseSpace.Infrastructure` 编译通过。
- 后端 `MuseSpace.Api` 编译通过。
- 前端 `muse-space-web` 执行 `npm run build` 通过。
- EF 迁移已被识别，并成功执行到数据库。

这意味着本次改动不仅停留在代码层面，而是已经完成了从模型、API、后台任务、前端页面到数据库的联动闭环。

## 8. 当前结果总结

经过这一轮改动后，原著导入功能相比之前有几个关键变化：

- 用户现在可以看到真实的上传进度和后台处理进度。
- 导入状态在刷新页面后可以恢复，不再依赖单次会话内存状态。
- Unicode 边界导致的切片写库异常被从根因上修正。
- Embedding 阶段从逐条处理升级为批量处理，导入效率明显更合理。
- 任务失败时，后端和前端都能看到更明确的错误信息。
- 后续如果继续做重试、管理页、性能分析，这一轮的状态模型已经足够支撑。

## 9. 后续建议

虽然这一轮已经把“原著导入可用性”和“可观察性”抬到了一个更稳的水平，但这个功能仍然有继续优化的空间。建议按优先级推进。

### 9.1 优先建议

- 把切片阶段也改成真正的增量进度，而不是当前“切片完成后一次性给出总数”。目前向量化已经是实时进度，切片阶段仍然偏粗。
- 增加导入失败后的“重试”入口，而不只是展示失败状态。现在已经具备状态和错误持久化，补重试会比较自然。
- 做一页更明确的导入日志或详情视图，把文件大小、切片数、开始结束时间、错误原因、总耗时直接展示出来。

### 9.2 性能建议

- 进一步把上传阶段从“先读进内存再存储”改为真正的流式写入和流式哈希计算，避免较大文件完全占用一块内存缓冲。
- 对导入耗时做指标采集，至少区分上传耗时、切片耗时、向量化耗时、数据库写入耗时，避免后续优化靠感觉推进。
- 如果后续用户会频繁导入更大的作品，可以考虑把原文件存储逐步切到对象存储，并评估是否需要把重型任务再拆到独立 Worker。

### 9.3 体验建议

- 在导入列表或详情页中增加“当前阶段说明”，例如“正在切片”“正在生成向量索引”“索引完成，可用于续写参考”。
- 如果导入文件较大，可以在上传前提示“上传后系统仍需要进行切片和索引，请耐心等待”。
- 对常见失败类型给出更友好的提示文案，例如编码异常、重复导入、外部向量服务失败、数据库写入失败等。

## 10. 结论

这次改动不是单纯把上传速度做快一点，也不是只给页面加一个进度条，而是围绕“导入作品”这件事，把稳定性、进度感知、错误定位和后续可扩展性一起补上了。

如果把这轮工作的价值概括成一句话，就是：

导入功能从“后台跑一串任务，用户只能等结果”，升级成了“有明确阶段、有真实进度、有失败定位、并且性能瓶颈已经开始被正面对付”的一条完整业务链路。
