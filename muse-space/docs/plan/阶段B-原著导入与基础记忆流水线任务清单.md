# 阶段B：原著导入与基础记忆流水线任务清单

> 阶段目标：打通“原著导入 -> 切片 -> 向量化 -> 检索 -> 状态反馈”的完整初级记忆闭环。
>
> 当前状态：核心后端链路已基本落地，前后端主要功能均已有代码锚点，建议转入补验收、补测试、补边界处理阶段。

---

## 1. 阶段范围

- 文件上传、去重、存储
- Chunk 生成
- Embedding 生成
- 向量表入库和索引
- 基础检索服务
- SignalR / 轮询状态反馈
- 前端导入与状态页

---

## 2. 可执行任务清单

### B-1 上传入口与重复导入控制

目标：建立稳定的原著导入入口。

任务：

- 确认上传接口支持 TXT / Markdown
- 确认 SHA256 去重策略覆盖同项目重复导入
- 明确 `novels.status` 状态流转
- 明确删除小说时是否级联删除 chunks 和 embeddings

实现思路：

- 上传请求只负责“接收、落库、投递任务”，不要在请求内同步做切片和 embedding
- 状态机必须足够简单，优先使用 `Pending -> Processing -> Indexed -> Failed`

验收：

- 重复文件导入不会生成重复任务
- 上传大文件时接口不会长时间阻塞

状态建议：已基本完成，建议补状态流转测试和重复上传回归测试。

### B-2 文本切片能力稳定化

目标：把文本切成可用于检索的稳定 chunk。

任务：

- 固定 chunk 大小和 overlap 参数
- 优先自然段、句号边界切片
- 处理空行、超长段、异常编码文本
- 补充单元测试，覆盖中文长文本和异常边界

实现思路：

- 切片优先关注“召回质量 + 可维护性”，不是追求最复杂语义切分
- 只要 chunk 在续写场景下足够稳定，就不急于引入 tokenizer 级精调

验收：

- 长文本 chunk 数量、offset、内容边界可预测
- 切片结果可稳定落库

状态建议：已完成主逻辑，建议补测试和参数说明。

### B-3 后台任务链路稳定化

目标：保证切片和向量化任务可追踪、可恢复、可重试。

任务：

- 确认 `ChunkNovelJob` 与 `EmbedNovelJob` 的串联关系
- 确认失败重试策略和错误记录方式
- 确认任务幂等性，防止重复执行造成脏数据
- 确认重启后任务恢复行为

实现思路：

- Hangfire 已足够承接当前复杂度，不需要此时拆独立 Worker
- 若后续 embedding 任务量放大，再在阶段E考虑把任务执行迁移出去

验收：

- 任意一个 Job 失败时，状态可见，且可重新执行

状态建议：已基本完成，建议补失败重试与幂等校验。

### B-4 Embedding 与向量表闭环

目标：让 chunk 可被高效语义检索。

任务：

- 确认 `IEmbeddingClient` 的默认实现和维度配置
- 确认 `memory.chunk_embeddings` 表、索引和模型名记录方式
- 确认嵌入生成后 `is_embedded` 或等效状态正确更新
- 补充向量维度不匹配时的保护逻辑

实现思路：

- 现阶段只需要单模型稳定产出，不要提前做多模型混合检索
- 记录 `model_name` 是为了将来重建向量，不是为了现在就做版本矩阵

验收：

- 任意导入完成后都能查到对应 embeddings
- 检索时模型与向量维度一致

状态建议：已基本完成，建议补一次从空库开始的全链路回归。

### B-5 基础检索服务

目标：给生成链路和后续 Agent 提供统一的原著检索入口。

任务：

- 确认 `INovelMemorySearchService` 的接口边界
- 确认检索时按项目隔离，不跨项目串数据
- 明确 topK 默认值和返回结构
- 增加日志，便于后续分析召回效果

实现思路：

- 阶段B只要先把“语义检索可用”做扎实
- 不要在这个阶段提前混入太多排序因子，否则难定位召回问题

验收：

- 输入一段场景描述，能返回相关 chunk 列表

状态建议：已完成主逻辑，建议增加检索质量评估样例。

### B-6 前端导入体验

目标：让原著导入是一个可见、可信、可恢复的用户流程。

任务：

- 确认 `novels.ts` API 封装完整
- 确认 `useNovelImportProgress` 正常接收 SignalR 推送
- 确认页面可展示 Pending / Processing / Indexed / Failed 状态
- 失败时提供轮询兜底与人工刷新入口

实现思路：

- 前端应展示真实状态，而不是假进度条
- 对于后台任务型流程，用户需要知道现在处于哪个阶段、为什么失败、能否重试

验收：

- 上传后用户可持续观察处理进度
- SignalR 失效时页面仍能通过状态接口恢复显示

状态建议：已基本完成，建议补异常状态和断线重连验证。

## 4. 本阶段不该过早引入的高级能力

以下能力在阶段B不应作为主线任务：

- Hybrid RAG 作为默认检索策略
- GraphRAG
- 复杂摘要式 memory items 自动抽取
- 多模型 embedding 比较

原因：阶段B首先要证明“基础导入闭环稳定可用”，否则后续增强都站不住。

---

## 5. 高级能力的真正引入时机

### Hybrid RAG

引入时机：阶段C中段开始评估，阶段C后段可进入默认主线。

具体步骤：

1. 先完成纯向量检索接入生成
2. 观察专有名词、角色名、设定词召回是否不足
3. 再补关键词检索和 RRF 融合
4. 最后再决定是否替换为默认检索策略

### 自动摘要记忆单元

引入时机：阶段E前段。

判断条件：chunk 已经太多，单靠原文片段回填 Prompt 导致上下文成本过高。
