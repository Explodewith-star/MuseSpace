# 阶段C：记忆接入生成主链路任务清单

> 阶段目标：把阶段B产出的检索能力真正接入续写、番外、草稿生成主链路，形成可感知的生成质量提升。
>
> 当前状态：关键代码锚点已存在，主链路已具备接入条件，建议将本阶段定义为“已完成主干，待继续优化召回和注入策略”。

---

## 1. 阶段范围

- `StoryContextBuilder` 注入检索结果
- Prompt 模板接入 `novel_context`
- 结构化上下文与文本记忆的双源融合
- 检索结果格式化和裁剪策略
- 生成质量评估与回归机制
- 为大纲规划与章节规划提供稳定输入基础

---

## 2. 可执行任务清单

### C-1 检索调用接入上下文构建器

目标：让每次生成都能按项目和场景目标动态检索原著记忆。

任务：

- 确认 `StoryContextBuilder` 已接入 `INovelMemorySearchService`
- 明确检索 query 来源，优先来自 scene goal 或用户当前生成意图
- 明确当检索为空时的降级路径
- 增加日志，记录每次生成的召回条数与 topK

实现思路：

- 检索结果属于生成上下文的一部分，而不是独立旁路
- 生成链路必须在检索失败时仍然能继续跑，避免记忆系统反向拖垮生成体验

验收：

- 同一个项目在不同 scene goal 下会得到不同的原著引用片段

状态建议：已基本完成，建议补日志和空结果回归测试。

### C-2 Prompt 模板融合策略

目标：把原著记忆安全地送进 Prompt，而不是简单拼接长文本。

任务：

- 确认 Prompt 模板已有 `novel_context` 段落
- 控制注入条数和单段长度，避免占满上下文窗口
- 统一引用格式，区分“原著参考”与“角色/规则摘要”
- 明确当结构化上下文与原著片段冲突时的提示优先级

实现思路：

- 原著片段的作用是“提供参考语境”，不是替代当前章节目标
- Prompt 中必须明确告诉模型：优先保持当前任务目标，同时参考原著风格和上下文

验收：

- 生成结果能够体现原著信息，但不会被引用片段强行带跑偏

状态建议：已完成主干，建议补长度治理策略。

### C-3 双源融合排序优化

目标：让结构化知识和文本记忆在同一次生成中协同工作。

任务：

- 明确结构化上下文和文本上下文的组合顺序
- 定义优先级，例如 `WorldRule > Character State > Novel Snippets > Chapter Summary`
- 对上下文总长度设预算，防止 Prompt 膨胀
- 记录哪些上下文片段最终进入了 Prompt，便于调优

实现思路：

- 这一阶段的关键不是“多”，而是“信息有层次”
- 结构化上下文负责硬约束，原著片段负责软约束和语义参考

验收：

- 同样的生成请求，打开记忆注入后结果更加贴近项目已有设定

状态建议：部分完成，建议补上下文预算策略和调试日志。

### C-4 生成质量回归与评估样例

目标：避免每次调检索策略都靠主观感觉判断效果。

任务：

- 建立最小评估样例集，覆盖角色一致性、设定引用、原著风格贴近度
- 固化 5 到 10 个典型续写 prompt
- 记录启用记忆前后的输出差异
- 形成可重复回归的人工评估模板

实现思路：

- 阶段C还不需要复杂自动评分体系
- 但必须先把“好坏如何判断”固定下来，否则 Hybrid 和更高级记忆优化无法评估

验收：

- 每次改检索策略后，至少能用同一批 case 做回归比较

状态建议：待补。

### C-5 为大纲规划提供输入基线

目标：让后续大纲规划 Agent 能基于可信项目资产生成结构，而不是只根据一句话自由发挥。

任务：

- 为大纲规划整理统一输入，包括项目摘要、角色摘要、世界观约束、文风要点、现有章节进度。
- 明确续写、番外、局部补全三类规划请求各自默认使用哪些上下文来源。
- 设计“规划查询”与“生成查询”的区别，避免把草稿生成的检索方式直接套给大纲规划。
- 记录参与大纲规划的上下文片段和约束集合，便于后续调优和追溯。

实现思路：

- 阶段C不直接承载完整的大纲 Agent，而是先把“规划所需上下文”沉淀清楚。
- 大纲规划应优先使用摘要化约束与结构化资产，原著片段作为证据和补充，而不是主输入。

验收：

- 面向同一个项目，续写规划与番外规划能够得到不同但有依据的上下文输入集合。

状态建议：待补，是阶段D大纲规划 Agent 的前置条件。

## 4. 高级能力引入时机

### 本阶段的技术边界

阶段C默认不新增以下能力作为主线前置：

- 对象存储
- Redis
- 独立 Worker
- Neo4j
- GraphRAG

原因：阶段C解决的是“原著记忆和结构化资产是否正确进入生成与规划输入”，而不是“先把系统做成完整工业化架构”。只要 `IStorageService`、检索接口和上下文构建边界清晰，后续替换底层实现即可。

### Hybrid RAG

引入时机：阶段C后半段，这是第一个真正适合引入的高级能力。

具体步骤：

1. 先确认纯向量检索主链路已稳定
2. 针对召回问题补一条关键词检索链路
3. 用 RRF 融合两路结果
4. 做固定 case 回归
5. 只有在结果稳定提升后，才把 Hybrid 设为默认

适合解决的问题：

- 人名、地名、招式名、组织名等专名召回差
- 规则条款、设定短句、术语片段匹配不准

### 重排序与时间衰减

引入时机：阶段C末尾到阶段E前段之间。

判断条件：基础 Hybrid 已稳定，但召回结果开始出现“相关但不够重要”的噪声。

### GraphRAG

引入时机：不在阶段C引入。

原因：阶段C的目标是把记忆接进生成，而不是重建知识图谱系统。

### 对象存储 / Redis / Worker

引入时机：不在阶段C默认引入。

原因：阶段C的关键问题是“上下文是否正确接入生成与规划”，不是“先把基础设施做重”。
