# 原著续写与生成质量深化方案

> 定位：在不做大规模重构的前提下，围绕 MuseSpace 当前最关键的 5 个体验缺口，补齐“原著续写”“长篇一致性”“资产提取覆盖率”“局部参考增强”“Token 可观测性”五条主线。
>
> 适用范围：原著导入后的续写、支线番外、章节草稿生成、资产提取、生成成本看板。
>
> 文档目标：作为后续 4 个里程碑的主执行方案。该方案不是新架构重写，而是在现有 SceneDraftSkill、StoryContextBuilder、ExtractNovelAssetsJob、ChapterDraftJob、GenerationRecord 等能力之上做深化。

---

## 0. 本轮要解决什么

当前用户反馈已经非常集中，核心不是“功能没有”，而是“生成链路还不够像真正的小说续写系统”。

本轮聚焦以下 5 个问题：

| 问题 | 当前表现 | 目标状态 |
| --- | --- | --- |
| 1. 原著续写 / 支线番外 | 已有原著导入，但还不能稳定从原著末尾继续写，也不能明确选择某条故事线做支线 | 让用户可以基于原著末尾、某段时间线、某条人物线稳定续写或写番外 |
| 2. 资产提取覆盖不足 | 长篇小说常常只提取出 20-30 个角色，重要配角缺失较多 | 对百章级作品稳定提取 60-100 个主要角色与重要配角，并支持分层展示 |
| 3. 当前章节参考片段缺失 | 生成章节时无法临时注入“只影响这一章”的参考片段 | 允许用户输入临时参考片段，并指定参考维度，如情绪、对话、节奏、文风 |
| 4. 前后剧情重复与矛盾 | 已发生事件会被 AI 遗忘或重复改写 | 建立“已发生事实”与“不可重复事件”约束，生成前注入，生成后校验 |
| 5. Token 消耗不可见 | 后台已有 GenerationRecord / AgentRun，但用户侧看不到成本视图 | 提供按项目、章节、功能、模型维度的 Token 看板 |

一句话概括，本轮目标是：

把 MuseSpace 从“有很多 AI 功能的写作工具”，推进成“基于原著世界持续创作的小说生产系统”。

---

## 1. 总体原则

### 1.1 不重构主架构，只深化主链路

本轮不推翻当前 Clean Architecture，不重写 Agent 系统，不重做前后端路由。

继续复用现有能力：

- `SceneDraftSkill` 作为章节生成核心技能。
- `StoryContextBuilder` 作为上下文装配入口。
- `ChapterDraftJob` 作为长任务编排主入口。
- `ExtractNovelAssetsJob` 作为原著导入后的资产抽取入口。
- `NovelMemorySearchService` 作为原著检索底座。
- `GenerationRecord` / `AgentRun` 作为生成与成本可观测基础。

### 1.2 结构化事实优先于“把 Prompt 塞更长”

长篇续写的问题，本质不是 Prompt 不够长，而是“已发生事实没有被稳定结构化”。

因此本轮优先建设：

- 章节事件时间线
- Canon Facts（固定事实）
- 角色关系状态
- 不可重复事件标记

而不是一开始就上 GraphRAG、Neo4j、复杂多 Agent 编排。

### 1.3 生成控制分成三层

每次章节生成，后续统一拆成三层控制：

1. 生成前：确定模式、检索范围、事件事实、参考片段。
2. 生成中：Prompt 明确区分“硬边界”“软参考”“原著召回”。
3. 生成后：自动检查是否重复事件、是否矛盾、是否越界。

### 1.4 可观测必须和生成能力同步建设

所有新增能力都要带可观测字段，不接受“先做功能，之后再补日志”。

至少记录：

- 使用模型
- Prompt Tokens / Completion Tokens / Total Tokens
- 检索到多少原著块
- 注入了多少角色 / 规则 / 事件
- 是否触发一致性告警

## 2. 现有能力与本轮新增能力的关系

### 2.1 当前可复用底座

当前系统已具备：

- 原著导入、切片、Embedding、向量检索
- 角色 / 世界观 / 文风提取与建议中心
- 章节规划、章节生成、批量生成
- 一致性检查、伏笔追踪、项目摘要
- AgentRun、GenerationRecord、FeatureFlag、Hangfire、SignalR

这些底座足够支撑本轮深化，不需要先推翻再重来。

### 2.2 本轮新增能力图

本轮建议新增 5 个能力模块：

| 模块 | 作用 | 核心落点 |
| --- | --- | --- |
| A. 续写模式层 | 区分原创、原著续写、支线番外、扩写 | 生成请求、UI 入口、上下文装配 |
| B. 局部参考层 | 为当前章节提供临时参考片段 | 生成表单、Prompt 参数 |
| C. Canon 事实层 | 固化已发生事件与关系状态 | 事件抽取、事实表、生成前注入、生成后校验 |
| D. 资产深化层 | 提高角色提取覆盖率和质量 | ExtractNovelAssetsJob 多阶段流水线 |
| E. 成本可视层 | 展示生成消耗和模型成本 | GenerationRecord / AgentRun 扩展、前端看板 |

---

## 3. 方案总览

```text
原著导入
→ 候选资产抽取与分层
→ 用户选择创作模式（原创 / 续写 / 支线 / 扩写）
→ 选择当前章节参考片段（可选）
→ StoryContextBuilder 组装：
   项目资产 + 原著检索 + 已发生事件 + Canon Facts + 支线约束 + 局部参考
→ SceneDraftSkill 生成章节
→ 生成后自动执行：
   重复事件检查 + 矛盾检查 + 越界检查 + Token 记录
→ 用户查看草稿、告警、成本，决定采纳 / 重写 / 修复
```

## 4. 模块 A：原著续写 / 支线番外模式

### 4.1 目标

让“原著导入”真正进入创作主链路，而不是只作为风格参考和资产来源。

用户应该可以明确表达：

- 接在原著最后一章之后继续写。
- 从原著某一时间点开始写支线。
- 围绕某几个人物写番外。
- 在不改原著结局的前提下补写隐藏故事。

### 4.2 模式设计

新增统一生成模式 `GenerationMode`：

| 模式 | 说明 |
| --- | --- |
| `Original` | 纯原创模式，按当前项目大纲生成 |
| `ContinueFromOriginal` | 接在导入原著的已知结尾后继续写 |
| `SideStoryFromOriginal` | 从原著中的某一人物线、章节范围或事件点生成支线番外 |
| `ExpandOrRewrite` | 对已有章节或参考片段做扩写 / 改写 |

### 4.3 后端设计

建议扩展当前章节生成请求，新增字段：

- `GenerationMode`
- `SourceNovelId`
- `ContinuationStartChapterNumber?`
- `OriginalRangeStart?`
- `OriginalRangeEnd?`
- `RelatedCharacterIds[]`
- `BranchTopic?`
- `DivergencePolicy`

其中 `DivergencePolicy` 建议支持：

- `StrictCanon`：不得改写原著既定事实
- `SoftCanon`：允许局部补写，但不得推翻关键设定
- `AlternateTimeline`：允许偏离，明确标记为平行线

### 4.4 上下文装配策略

`StoryContextBuilder` 在续写模式下，不能只取“最近章节摘要”，而要补 4 类上下文：

1. 原著结尾摘要：用于“接续写”。
2. 原著指定章节范围摘要：用于“支线番外”。
3. 相关人物状态快照：关系、身份、未解决目标。
4. 不可改写事实：已确认的婚约、死亡、身份暴露、世界规则突破等。

### 4.5 前端交互

在章节生成入口增加“创作模式”卡片：

- 生成模式下拉
- 原著范围选择
- 相关人物多选
- 时间点说明
- 是否允许偏离原著

对于 `SideStoryFromOriginal`，至少要有两个显式选择：

- 围绕谁写
- 发生在原著哪一段

否则用户虽然写的是“番外”，模型拿到的仍然是模糊命令。

### 4.6 验收标准

- 用户可从“原著末尾继续写”生成一章，且明显承接原著结尾状态。
- 用户可指定某人物线生成支线番外，且不误拉入无关主线事件。
- 在 `StrictCanon` 下，不允许生成与原著冲突的关键事件。

## 5. 模块 B：当前章节参考片段

### 5.1 目标

允许用户在生成某一章时，临时注入一段参考片段，只影响当前章节，不写入长期资产，不污染 Canon 事实。

### 5.2 能力定义

新增“当前章节参考片段”输入区，支持：

- 参考片段正文
- 参考目的类型
- 参考强度

建议参考目的类型：

| 类型 | 说明 |
| --- | --- |
| `Emotion` | 参考情绪氛围 |
| `Dialogue` | 参考对话节奏与说话方式 |
| `NarrativeRhythm` | 参考叙事快慢和段落推进 |
| `StyleTexture` | 参考语言质感、描写浓度 |
| `SceneStructure` | 参考这一场戏如何起承转合 |
| `InteractionTension` | 参考人物互动张力 |

参考强度建议：

- `Low`
- `Medium`
- `High`

### 5.3 Prompt 规则

Prompt 中必须明确区分：

- `硬边界`：本章目标、已发生事件、不可改写事实
- `软参考`：当前章节参考片段

建议在模板中加入明确规则：

> 参考片段仅用于理解本章所需的情绪、对话方式、节奏或表达方向。不得复述、改写、照搬其中的具体句子、桥段、结构或独特表达。若参考片段与本项目既定事实冲突，以本项目事实为准。

### 5.4 数据边界

这类参考默认只作用于单次生成请求：

- 不写入 `StoryContext` 持久状态
- 不写入角色 / 世界观 / 文风资产
- 不参与后续章节事实抽取

### 5.5 验收标准

- 用户可以给单章粘贴参考片段并指定“主要参考情绪”。
- 输出文本在氛围和表达方向上向参考靠拢，但不会复制原句。
- 去掉参考片段后，同一章输出风格明显变化。

## 6. 模块 C：长篇一致性与 Canon Ledger

### 6.1 目标

解决“第三章已经求婚成功，第六章又重新求婚且结果相反”这一类前后重复与矛盾问题。

这不是简单的“检索更多文本”问题，而是缺少结构化事实层。

### 6.2 建议新增的两层结构化记忆

#### 第一层：章节事件 `chapter_events`

用于记录每一章发生过什么，偏“时间线”。

建议字段：

- `Id`
- `ProjectId`
- `ChapterId`
- `Order`
- `EventType`
- `EventText`
- `ActorCharacterIds[]`
- `TargetCharacterIds[]`
- `Location?`
- `TimePoint?`
- `Importance`
- `IsIrreversible`
- `CreatedAt`

#### 第二层：固定事实 `canon_facts`

用于记录“已经成为既定事实”的内容，偏“状态真相”。

建议字段：

- `Id`
- `ProjectId`
- `FactType`
- `SubjectId?`
- `ObjectId?`
- `FactKey`
- `FactValue`
- `SourceChapterId`
- `Confidence`
- `IsLocked`
- `InvalidatedByChapterId?`
- `CreatedAt`

可覆盖的事实类型示例：

- 关系状态：已恋爱、已订婚、已决裂
- 身份状态：已暴露、未暴露、失忆、恢复记忆
- 生死状态：存活、死亡、失踪
- 世界状态：规则已被打破、力量已觉醒
- 不可重复事件：求婚已发生、退婚已发生、决斗已发生

### 6.3 生成前注入

`StoryContextBuilder` 在生成第 N 章时应额外注入：

- 最近 3 章详细事件
- 更早章节的关键事件摘要
- 当前关键人物的状态快照
- 本章相关的不可重复事件列表

Prompt 里要明确分出：

- `## 已发生事件时间线`
- `## 当前人物关系与状态`
- `## 不可重复 / 不可改写事实`

### 6.4 生成后校验

在 `ChapterDraftJob` 后处理链中新增两类守护：

1. `DuplicateEventCheckJob`
   - 检查本章是否重复发生已标记不可重复的事件。

2. `CanonConflictCheckJob`
   - 检查本章是否否认、反转、遗漏已锁定事实。

输出继续落入建议中心，但建议严重级别至少区分：

- `Warning`
- `Blocking`

其中 `Blocking` 可在后续决定是否禁止“一键采纳为定稿”。

### 6.5 用户可编辑兜底

AI 自动抽取不会 100% 正确，因此必须给用户一个“修事实”的入口。

建议在章节详情页新增两个区块：

- 事件清单
- Canon 事实清单

支持用户手动：

- 新增事件
- 修正文案
- 标记该事件不可重复
- 锁定 / 解锁某个事实

### 6.6 验收标准

- 构造“第三章求婚成功，第六章重复求婚”的样例，系统能在生成后检出。
- 在生成第六章时，Prompt 中能稳定看到“已订婚”或“求婚已发生”的硬事实。
- 用户手动修正事实后，后续生成能明显遵循修正结果。

## 7. 模块 D：资产提取覆盖率提升

### 7.1 目标

从当前“只能提取出 20-30 个角色”，提升到“对百章级作品稳定产出 60-100 个主要角色与重要配角候选”。

注意，这里的目标不是“把所有路人角色全部做成卡片”，而是：

- 主角必须全
- 重要配角尽量全
- 次要配角可分层保留，不强行全部上墙

### 7.2 当前短板

当前 `ExtractNovelAssetsJob` 更接近“抽样 + 单轮提取”，因此天然会偏向：

- 高频人物
- 前期角色
- 主线角色

对以下对象容易漏：

- 中后期重要角色
- 低频但关键角色
- 多别名人物
- 群像小说中的支线核心人物

### 7.3 建议改造成四阶段流水线

#### D1. 分块候选提取

不是只从少量 chunk 里提人物，而是按全书 chunk 扫描候选人物：

- 名称
- 别名
- 初次出现位置
- 最近出现位置
- 关联人物
- 行为与身份线索

输出进入候选池，而不是直接建角色卡。

#### D2. 合并去重

针对以下情况做归并：

- 同名异写
- 昵称 / 尊称 / 本名
- 姓氏指代
- 角色身份变更前后名称不同

#### D3. 重要性评分

对候选人物计算一个综合分：

- 出现频次
- 分布跨度
- 参与关键事件次数
- 与主角的关联强度
- 是否推动情节转折

按分数分层：

- `Core`
- `MajorSupporting`
- `MinorSupporting`
- `Incidental`

#### D4. 二次补全

只对 Top N 角色做更贵的补全抽取，例如：

- 性格
- 动机
- 说话方式
- 秘密
- 当前状态
- 与主角关系

### 7.4 展示与采纳策略

建议中心不要把所有候选都混在一起。

建议按层展示：

- 已识别主要角色
- 已识别重要配角
- 次要角色候选

这样用户既能看到覆盖率提升，也不会被几百个候选淹没。

### 7.5 执行建议

`ExtractNovelAssetsJob` 建议拆为：

- `ExtractCharacterCandidatesJob`
- `MergeCharacterCandidatesJob`
- `RankCharacterCandidatesJob`
- `EnrichTopCharactersJob`

也可以先内部拆函数、外部仍保留一个总 Job 名称，避免一次性改动过大。

### 7.6 验收标准

- 对一部长篇作品，候选人物池能稳定产出远高于当前版本的数量。
- Top 60-100 角色中，主角和主要配角的缺失率明显下降。
- 用户在建议中心能区分“主角色建议”和“次要候选”。

## 8. 模块 E：Token 消耗看板

### 8.1 目标

让用户和开发者都能回答这些问题：

- 哪个功能最费 Token？
- 哪一章生成最贵？
- 哪个模型性价比最好？
- 一次续写到底消耗了多少上下文？

### 8.2 数据模型建议

在现有 `GenerationRecord` / `AgentRun` 基础上补齐至少以下字段：

- `PromptTokens`
- `CompletionTokens`
- `TotalTokens`
- `EstimatedCost`
- `Provider`
- `ModelName`
- `FeatureName`
- `ChapterId?`
- `NovelId?`
- `GenerationMode?`
- `RetrievedChunkCount`
- `InjectedCharacterCount`
- `InjectedEventCount`
- `InjectedCanonFactCount`

### 8.3 看板视图

建议先做三层视图：

#### 项目级

- 今日 / 本周 / 本月消耗
- 按功能分布
- 按模型分布

#### 章节级

- 某章生成历史
- 单次生成耗时、Token、告警情况
- 修复重写累计成本

#### 管理级

- 所有项目成本排行
- 模型成本占比
- 平均成功率 / 平均单次成本

### 8.4 前端入口

建议新增两个轻量入口：

- 项目概览页的“本项目 AI 消耗”卡片
- 章节详情页的“本章生成记录”抽屉

管理员页后续再补完整统计页。

### 8.5 验收标准

- 生成一章后，用户可立即看到本次 Token 使用量。
- 项目概览中能看到最近 7 天按功能分组的消耗。
- 管理员能按模型维度查看总成本与成功率。

## 9. 推荐实施顺序

本轮不建议五项一起并行硬上，推荐按“高收益、低改动”到“高价值、重工程”推进。

### Milestone 1：局部参考 + Token 可视化（4-6 天）

目标：先把用户最能感知的质量提升和成本透明度做出来。

包含：

- 模块 B：当前章节参考片段
- 模块 E：Token 看板基础版

交付后用户立即能感知：

- 某一章更好调教
- 每次生成不再是黑箱

### Milestone 2：Canon 事实层（一致性根治，6-10 天）

目标：先解决“重复求婚”这类长篇硬伤。

包含：

- `chapter_events`
- `canon_facts`
- 生成前注入
- 生成后重复 / 冲突检查

这一阶段做完后，再推进原著续写，成功率才会显著更高。

### Milestone 3：原著续写 / 支线番外模式（5-8 天）

目标：让原著导入真正进入主创作链路。

包含：

- `GenerationMode`
- 原著范围选择
- 支线人物线选择
- `DivergencePolicy`

### Milestone 4：资产提取流水线升级（7-12 天）

目标：提升长期创作的素材覆盖度和角色资产完整性。

包含：

- 候选池
- 去重归并
- 排名分层
- TopN 补全

之所以放在最后，不是因为它不重要，而是因为前 3 个里程碑对“生成体验”改善更直接，且能反过来帮助确定角色提取的真实阈值需求。

## 11. 风险与边界

### 11.1 本轮明确不做

| 项 | 原因 |
| --- | --- |
| GraphRAG / Neo4j | 当前问题仍可通过章节事件 + Canon Facts 解决，没必要提前引入复杂度 |
| Redis / 独立 Worker | 当前任务规模尚可由 Hangfire 承担 |
| 多 Agent 自治写整本小说 | 现在更需要控制力和事实稳定性，而不是更复杂的自治 |
| 把所有角色都做成完整角色卡 | 成本高、收益低，本轮目标是“主角 + 重要配角充分覆盖” |

### 11.2 主要风险

| 风险 | 应对 |
| --- | --- |
| 事实抽取误判 | 保留人工编辑事件 / 事实入口 |
| 参考片段被模型过度模仿 | Prompt 明确禁止照搬，并加入后处理抽查 |
| 角色候选爆炸导致建议中心难用 | 做分层展示，不把所有候选平铺 |
| 统计字段缺失导致看板不准 | 在所有 LLM 调用链路统一记录 Tokens，先把基础字段补齐 |

## 12. 最终验收口径

如果本方案执行完成，产品层面应至少出现以下变化：

1. 用户能明确以“原著续写”或“支线番外”模式发起章节生成。
2. 用户能为某一章临时注入参考片段，并指定参考方向。
3. 系统能明显减少“已发生事件重复发生”与“人物关系倒退”的问题。
4. 长篇作品的人物提取结果覆盖率显著高于当前版本，主角和重要配角缺失明显下降。
5. 用户能看到每次生成具体用了多少 Token、用了哪个模型、消耗集中在哪些功能。

---

## 13. 结论

这不是一轮“再加几个 Agent”的工作，而是一轮把 MuseSpace 创作主链路从“会生成”推进到“会持续创作、会自我约束、可被用户调教、成本透明”的系统化升级。

推荐执行顺序：

1. 当前章节参考片段
2. Token 看板
3. Canon 事实层
4. 原著续写 / 支线番外模式
5. 资产提取流水线升级

这样推进，既不会打散现有架构，也能让每一个里程碑都直接改善用户实际体验。
