# 阶段性完善计划

> 定位：阶段 C / D 主干已完成、阶段 E 工业化扩展尚不必启动的中间过渡。
>
> 把"用户体验"与"生成质量"两条线分开管理，按里程碑独立交付。
>
> 详细任务编号与现状锚点见 [`阶段C+一致性补强与创作闭环任务清单.md`](./阶段C+一致性补强与创作闭环任务清单.md)，本文件聚焦**开发计划与执行顺序**。

---

## 一、计划划分

本阶段分为两大独立模块，各自有清晰的交付价值和验收路径：

| 部分 | 模块 | 用户感知 | 工期 |
|---|---|---|---|
| **第一部分：创作工作流优化与完善** | A1 草稿一键采用为定稿 / A2 定稿一键导出 / A3 批量章节草稿生成 | 不再手动复制粘贴；能直接导出阅读；多章一键生成 | 4-6 天 |
| **第二部分：解决"一致性"问题** | B1 Prompt 边界硬化 / B2 章节事件时间线 / B3 草稿越界反向校验 | AI 不再写大纲外事件；前后矛盾消失；越界自动报警 | 6-9 天 |

> 两部分之间**没有强依赖**，可以并行或交错推进；但批量生成（A3）建议放在 B1 / B2 之后，避免越界被批量放大。

---

## 二、第一部分：创作工作流优化与完善

> 目标：让"AI 生成草稿 → 采纳为定稿 → 导出整本"成为闭环，且支持一键批量生成多章。

### A1 草稿一键采用为定稿（0.5 天）

对应任务清单：C+4.1 / C+4.2 / C+4.3。

**问题**：当前 AI 生成草稿后，用户必须手动复制粘贴到定稿输入框再保存。

**方案**：

- 后端：`POST /api/storyprojects/{projectId}/chapters/{chapterId}/adopt-draft`
  - `FinalText = DraftText`，`Status → Final`，**保留** `DraftText` 用于对照。
  - 若 `FinalText` 已有内容且 `overrideExisting != true` → 返回 409，前端弹确认框。
- 前端：草稿卡片头新增"采用为定稿"按钮；定稿非空时弹二次确认（显示"原定稿 X 字 → 草稿 Y 字"）。

### A2 定稿一键导出（1-2 天）

对应任务清单：C+6.1 ~ C+6.5。

**问题**：用户写完想看整本小说时，需要逐章复制定稿。

**方案（v1 只做 Markdown + TXT，零依赖）**：

- 后端：
  - 新增 `IChapterExportService`，按 `ChapterExportOptions(Format / FromNumber / ToNumber / IncludeDraftFallback / OnlyFinal)` 拼接。
  - Markdown：`# 《项目名》` + 每章 `## 第 N 章 标题` + 章节正文 + `---` 分隔。
  - TXT：每章 `第 N 章 标题\n\n` + 正文 + 空行。
  - **UTF-8 + BOM**（兼容 Windows 记事本中文）。
- 接口：`GET /api/storyprojects/{id}/export?format=md|txt&from=&to=&includeDraft=false&onlyFinal=true`
- 文件名：`《项目名》_第1-15章_20260430-1430.md`
- 前端：章节列表顶部 + 项目概览页各加一个"导出"按钮，弹窗包含格式 / 范围 / 是否包含草稿三项。
- `.docx` / `.epub` 暂不做：先用 md / txt 验证完整闭环。

### A3 批量章节草稿生成（2-3 天）

对应任务清单：C+5.1 ~ C+5.6。

**问题**：列了 15 章大纲后，用户需要逐章点"生成草稿"。

**核心边界判断**：

| 项 | 推荐值 | 理由 |
|---|---|---|
| 默认上限 | **5 章/批** | 5-15 分钟内完成，失败重做代价小 |
| 硬上限 | **10 章/批** | 10 章 ≈ 60K 输出 token、40 个后处理 Job 排队，已是临界 |
| 执行模式 | **顺序串行** | 第 N 章依赖第 N-1 章已抽取事件（B2 上线后）；规避 LLM 速率限制 |
| 单章超时 | 5 分钟 | — |
| 单批超时 | 30 分钟 | — |
| 失败容忍 | 单章失败不中断后续 | 记录到 `FailedChapterIds[]`，最终状态 `PartiallyFailed` |
| 可中止 | ✅ | 通过 `CancelRequested` 标志，正在跑的当前章节做完后中止 |

**方案**：

- 新表 `chapter_batch_draft_job_runs` 跟踪进度（幂等建表）。
- 三个接口：创建批次 / 查询进度 / 取消。
- `BatchChapterDraftJob` 顺序复用 `ChapterDraftJob.ExecuteAsync`（**直接调用，非 Enqueue**）。
- SignalR `NotifyBatchProgressAsync` 推送进度。
- 前端章节列表顶部"批量生成草稿"按钮 + 进度条（当前章节 / 已完成 / 失败 / 剩余 / 中止）。
- 三个 FeatureFlag：`feature.batch-draft.enabled` / `feature.batch-draft.max-size` / `feature.batch-draft.parallel-enabled`（保留位，默认 false）。

> **为什么放最后做？** A3 必须等 B1 / B2 沉淀后才能放心开放——否则 AI 越界会被批量放大，一次性生成 5 章错误比逐章生成更难修。

## 三、第二部分：解决"一致性"问题

> 目标：解决用户反馈的 "列了 15 章大纲，AI 在生成第 3 章时却写了大纲里没发生的事情" 问题。
>
> **不引入知识图谱 / GraphRAG / Neo4j**——这是 Prompt 边界问题，不是图谱召回问题。

### B1 Prompt 边界硬化（1-2 天，立竿见影）

对应任务清单：C+1.1 ~ C+1.5。

**根因**：
- `StoryContextBuilder` 当前按 `Number DESC` 取最近 3 章摘要——写中段章节时会把**后续未发生章节**摘要当成"已发生"塞进 Prompt。
- 章节信息全塞进 `SceneGoal` 字符串，模型难区分"本章目标"与"边界约束"。
- Prompt 中只有一句软规则"不要矛盾"，对模型约束力弱。

**方案**：

- **B1.1** [`StoryContextBuilder.cs`](../../src/MuseSpace.Infrastructure/Story/StoryContextBuilder.cs#L66-L73) 修正取章窗口：`StoryContextRequest` 加 `CurrentChapterNumber`，过滤 `Number < CurrentChapterNumber` 后再取最近 3 章。
- **B1.2** [`StoryContext.cs`](../../src/MuseSpace.Application/Abstractions/Story/StoryContext.cs) 新增 `CurrentChapterPlan / PreviousChapterRecap / NextChapterPreview` 三个字段。
- **B1.3** [`ChapterDraftJob.cs`](../../src/MuseSpace.Infrastructure/Jobs/ChapterDraftJob.cs#L62-L86) 拆分参数：边界字段单独经 `Parameters` 传递，不再全塞 SceneGoal。
- **B1.4** [`scene-v1.md`](../../prompts/drafting/scene-v1.md) 加三个强制段：`## 本章大纲（强制边界）` / `## 上一章已发生` / `## 下一章保留（不要在本章写）`，并加硬指令：
  > **严格禁止：写入本章大纲范围之外的事件、人物转折、关键情节。本章只能推进"本章大纲"中列出的目标，不得提前写入"下一章保留"中的内容，不得改写"上一章已发生"中的事实。**
- **B1.5** 原著片段段头加分区警告：
  > 以下来自**原著小说**，仅作为**风格 / 语境参考**，**并非本作已发生情节**。不得直接借用其中事件作为本章已发生事实。

**预期**：约 70% 的"AI 编造大纲外事件"消失。

**验收**：复用 [`C-4-草稿生成评估样例集.md`](./C-4-草稿生成评估样例集.md) 跑回归对比。

### B2 章节事件时间线（3-5 天，结构化记忆）

对应任务清单：C+2.1 ~ C+2.5。

**目标**：让 AI 在生成第 N 章时知道前 N-1 章具体发生过什么事，而不是只看模糊摘要。

**方案**：

- **B2.1** 新表 `chapter_events`：`Id / ChapterId / Order / EventText / Actors[] / Location? / TimePoint? / CreatedAt`。复用 `SchemaMigrationRunner.RunOnceAsync` 幂等建表。
- **B2.2** `ExtractChapterEventsJob`：在 `ChapterDraftJob` 后链式入队，把刚生成的草稿压成 3-8 条结构化事件写入 `chapter_events`。用最便宜模型即可。
- **B2.3** `StoryContextBuilder` 新增 `EventTimeline` 字段：取所有 `Number < N` 的事件，总字符预算 2000，超预算时**最近 3 章详细 + 远期摘要式**两段式。
- **B2.4** Prompt 新增 `## 已发生事件时间线` 段，并在规则中明确"视为已固定事实，不得改写或矛盾"。
- **B2.5** 章节详情页侧栏新增"事件清单"区块，支持手动编辑 / 补录 / 删除（修正 AI 抽取错误）。

**预期**：剩余 20%-25% 的"前后矛盾、人物状态错位"消失。

### B3 草稿越界反向校验（1-2 天，兜底守护）

对应任务清单：C+3.1 ~ C+3.4。

**目标**：即便 Prompt 失控也能事后扫出来推送给用户；同时为后续判断"是否真要上 GraphRAG"提供数据依据。

**方案**：

- **B3.1** 新增 `DraftOutlineComplianceAgent`，输出 JSON 数组：
  ```json
  { "type": "out-of-scope" | "contradicts-prev" | "skips-next-content",
    "excerpt": "...", "severity": "low|medium|high", "suggestion": "..." }
  ```
- **B3.2** 链入 [`ChapterDraftJob.cs`](../../src/MuseSpace.Infrastructure/Jobs/ChapterDraftJob.cs#L107-L114) 后处理序列，与 StyleConsistency / CharacterConsistency / PlotThread 并列。
- **B3.3** 复用 `OutlineConsistency` 类目沉淀到 `agent_suggestions`，章节页 / 大纲页通过现有 `PendingSuggestionPanel` 展示。
- **B3.4** FeatureFlag `feature.draft-outline-compliance.enabled`，便于灰度与紧急关闭。

## 四、实施排期

按"先治本 + 顺手做体验，后做批量"的原则切成三个里程碑：

```
┌─────────────────────────────────────────────────────────────────┐
│ Milestone 1（Day 1-5）：B1 边界 + A1 采纳 + A2 导出             │
│   交付：① 草稿不再越界  ② 不用复制粘贴  ③ 能导出阅读           │
├─────────────────────────────────────────────────────────────────┤
│ Milestone 2（Day 6-11）：B2 时间线 + B3 反向校验                │
│   交付：④ 前后矛盾消失  ⑤ AI 越界自动报警                      │
├─────────────────────────────────────────────────────────────────┤
│ Milestone 3（Day 12-14）：A3 批量生成                           │
│   交付：⑥ 一键生成多章                                         │
└─────────────────────────────────────────────────────────────────┘
```

### Milestone 1（Day 1-5）

| Day | 任务 |
|---|---|
| 1-2 | **B1** Prompt 边界硬化（B1.1 ~ B1.5）；Day 2 末跑一次评估样例集回归 |
| 3   | **A1** 草稿一键采用为定稿 |
| 4-5 | **A2** 定稿一键导出（md / txt） |

✅ **里程碑结束**：用户当晚能边界正确生成草稿 → 一键采纳为定稿 → 一键导出全本。

> 🚦 **决策点**：Milestone 1 完成后跑评估样例。**如果一致性问题已基本消失**，Milestone 2 可降级为可选——直接跳到 M3 批量生成。这是给项目省 6 天的关键决策点。

### Milestone 2（Day 6-11）

| Day | 任务 |
|---|---|
| 6-7 | **B2.1 + B2.2** `chapter_events` 表 + `ExtractChapterEventsJob` |
| 8   | **B2.3 + B2.4** 时间线注入 + Prompt 新段 |
| 9   | **B2.5** 章节详情页事件清单 UI |
| 10-11 | **B3** 草稿越界反向校验（B3.1 ~ B3.4） |

✅ **里程碑结束**：剩余前后矛盾问题消失，AI 越界自动检出并推到待办面板。

### Milestone 3（Day 12-14）

| Day | 任务 |
|---|---|
| 12 | **A3.1 + A3.2** `chapter_batch_draft_job_runs` 表 + 三个 REST 接口 |
| 13 | **A3.3 + A3.4** `BatchChapterDraftJob` 编排 + SignalR 进度推送 |
| 14 | **A3.5 + A3.6** 前端入口 / 进度条 / 中止按钮 + FeatureFlag |

✅ **里程碑结束**：阶段 C+ 整体收尾。

## 五、风险与缓冲

| 风险 | 应对 |
|---|---|
| LLM 提供商速率限制（批量生成时） | A3 顺序执行 + 单章 5 分钟超时已规避；不引入并发 |
| `chapter_events` Agent 抽取要点错误 | B2.5 提供手动编辑入口兜底 |
| B3 反向校验误报率高 | B3.4 FeatureFlag 灰度，必要时关闭 |
| Prompt 改写后老样例失效 | M1 收尾时跑评估样例做基线对比，M2 完成后再跑一次 |
| 批量生成中途失败 | 单章失败不中断后续，`FailedChapterIds[]` 留痕，用户可重做 |
| M2 工作量大于预期 | 利用 M1 决策点直接跳过 M2，先上 M3 |

## 六、与阶段 E 的衔接

| 本阶段完成后的观察 | 阶段 E 决策 |
|---|---|
| 一致性问题已基本消失 | 阶段 E 按原文档计划走（memory_items / 实体关系 / 工业化）|
| 跨多卷复杂关系仍频繁出错 | 阶段 E-5 GraphRAG 试点有真实需求依据，可启动 |
| 批量生成成为高频操作 | 阶段 E-4 独立 Worker 优先，把 `BatchChapterDraftJob` 拆出去 |

A3 的 `chapter_batch_draft_job_runs` 任务模型是为阶段 E-4 Worker 化预留的迁移对象——届时不必重新设计任务表。

## 七、不在本阶段做（明确边界）

| 项 | 原因 |
|---|---|
| Hybrid 检索 | 与本阶段问题无关；保留为阶段 C 末尾或阶段 E 前期独立任务 |
| memory_items / 实体关系 | 阶段 E 内容；先验证 B1/B2 是否够用 |
| GraphRAG / Neo4j | 阶段 E-5；除非本阶段完成后仍有跨多卷复杂关系问题 |
| Redis / 独立 Worker | 当前压力不需要；A3 顺序执行已能满足 |
| `.docx` / `.epub` 导出 | 增加依赖收益不明显，先用 md / txt 验证导出闭环 |
| 章节并发批量生成 | A3 已留 FeatureFlag 位，待真实需求出现再开 |
